## BUG #1: WebSocket Parameter Destructuring (CRITICAL)

### Severity: CRITICAL
### Impact: Blocks 12+ evasion commands
### Estimated Fix Time: 15 minutes
### Risk Level: LOW

### Problem

The WebSocket server destructures incoming messages incorrectly, causing all named parameters to be nested one level deeper than expected.

### Root Cause

In `/websocket/server.js` at line 8284:

```javascript
async handleCommand(data, options = {}) {
  const { command, ...params } = data;
  // ...
  const result = await handler(params);
}
```

When a client sends:
```javascript
{ 
  command: 'apply_fingerprint', 
  params: { profileId: 'fp_xxx' } 
}
```

The destructuring produces:
```javascript
command = 'apply_fingerprint'
params = { params: { profileId: 'fp_xxx' } }  // WRONG!
```

But handlers expect:
```javascript
params.profileId = 'fp_xxx'
```

Instead they receive:
```javascript
params.params.profileId = 'fp_xxx'
```

### Affected Commands (12 total)

1. `apply_fingerprint` - Cannot apply fingerprint to page
2. `configure_canvas_noise` - Cannot configure canvas evasion
3. `configure_webgl_noise` - Cannot configure WebGL evasion
4. `configure_audio_noise` - Cannot configure audio evasion
5. `configure_font_evasion` - Cannot configure font evasion
6. `set_evasion_levels` - Cannot set evasion levels
7. `get_evasion_config` - Cannot retrieve evasion config
8. `set_active_fingerprint` - Cannot set active profile
9. `generate_mouse_path` - Cannot generate mouse movements
10. `generate_typing_events` - Cannot generate typing simulation
11. `generate_scroll_behavior` - Cannot generate scroll simulation
12. `create_behavioral_profile` - May have issues with optional params

### Evidence from Testing

**Error Messages:**
```
apply_fingerprint: "No profile specified and no active profile"
  (Actually: params.profileId is undefined because params.params.profileId is the real value)

configure_canvas_noise: "No profile specified and no active profile"
  (Same root cause)

generate_mouse_path: "Cannot read properties of undefined (reading 'x')"
  (Actually: params.start is undefined because params.params.start is the real value)
```

### Solution

**Option A: Use data.params directly (Recommended)**

```javascript
// File: /websocket/server.js, line 8283-8312

async handleCommand(data, options = {}) {
  const { command, params } = data;  // Extract params explicitly
  const { enableRetry = true, maxRetries = ERROR_RECOVERY_CONFIG.maxRetries } = options;

  if (!command) {
    return { success: false, error: 'Command is required' };
  }

  const handler = this.commandHandlers[command];
  if (!handler) {
    // ... rest of error handling unchanged
  }

  let lastError = null;
  let attemptCount = 0;
  const canRetry = enableRetry && isRetryableCommand(command);

  while (attemptCount <= (canRetry ? maxRetries : 0)) {
    try {
      const result = await handler(params || {});  // Pass params correctly
      // ... rest unchanged
    } catch (error) {
      // ... rest unchanged
    }
  }
}
```

**Option B: Handle backward compatibility**

If there's a possibility that some clients send params differently:

```javascript
async handleCommand(data, options = {}) {
  const { command, ...rest } = data;
  // Support both { command, params: {...} } and { command, ...params }
  const params = rest.params || rest;
  
  // ... rest of implementation unchanged
}
```

## BUG #2: Scroll Behavior Returns Empty Events (MODERATE)

### Severity: MODERATE
### Impact: Scroll simulation cannot be validated
### Estimated Fix Time: 30 minutes
### Risk Level: MEDIUM

### Problem

The `generate_scroll_behavior` command succeeds but returns an empty events array with 0 duration.

### Current Behavior

```
Input:
  sessionId: session_1778276494723
  distance: 500
  direction: down

Output:
  success: true
  events: []
  eventCount: 0
  totalDuration: 0ms
```

### Expected Behavior

```
Output:
  success: true
  events: [
    { type: 'scroll', x: 500, y: 0, t: 0 },
    { type: 'scroll', x: 500, y: 50, t: 150 },
    { type: 'scroll', x: 500, y: 100, t: 320 },
    ...
  ]
  eventCount: 8-15
  totalDuration: 800-1200ms
```

### Root Cause

Check the `MouseMovementAI.generateScrollBehavior()` implementation:

**Likely Locations:**
- `/evasion/behavioral-ai.js` - MouseMovementAI class, `generateScrollBehavior` method
- `/websocket/commands/evasion-commands.js` - Command handler (line 828)

### Investigation Steps

```bash

# 2. Check if method exists

# 3. Verify handler calls it correctly
grep -B 5 -A 15 "generate_scroll_behavior" /home/devel/basset-hound-browser/websocket/commands/evasion-commands.js
```

### Solution Approach

1. Verify the method generates events (check for early return)
2. Ensure distance parameter is used correctly
3. Confirm direction parameter affects Y-axis changes
4. Verify timing follows human-like patterns

### Expected Implementation Pattern

```javascript
generateScrollBehavior(distance, direction = 'down') {
  const events = [];
  const totalDuration = this.calculateScrollDuration(distance);
  const steps = Math.floor(distance / 50) + Math.random() * 5;
  
  for (let i = 0; i < steps; i++) {
    const progress = i / steps;
    const t = totalDuration * progress;
    const scrollAmount = distance * progress;
    const yOffset = direction === 'down' ? scrollAmount : -scrollAmount;
    
    events.push({
      type: 'scroll',
      x: window.innerWidth / 2,
      y: window.innerHeight / 2 + yOffset,
      t: Math.floor(t)
    });
  }
  
  return events;
}
```

## BUG #3: Platform Type Not Correctly Set (MINOR)

### Severity: MINOR
### Impact: Platform-specific fingerprints may not match requested platform
### Estimated Fix Time: 20 minutes
### Risk Level: LOW

### Problem

When creating fingerprints with specific platforms (macos, linux), the returned profile shows `platformType: "windows"`.

### Evidence

```
Test Input: create_fingerprint_profile({ platform: 'macos' })

Test Output:
  profileId: fp_83a970223a1b0b7a
  profile.platformType: "windows"  // WRONG - should be "macos"
  profile.userAgent: Mozilla/5.0... // User agent seems correct

Test Input: create_fingerprint_profile({ platform: 'linux' })

Test Output:
  profileId: fp_9ec7b2c6514a25d5
  profile.platformType: "windows"  // WRONG - should be "linux"
```

### Root Cause

Check `/evasion/fingerprint-profile.js` FingerprintProfile constructor:

Likely issue is one of:
1. Platform parameter not being passed to constructor
2. Constructor not assigning platform parameter to this.platformType
3. Hardcoded value overwriting platform assignment

### Solution

In `/evasion/fingerprint-profile.js`, verify the constructor:

```javascript
class FingerprintProfile {
  constructor(options = {}) {
    const {
      id,
      platform = 'windows',  // DEFAULT VALUE
      timezone = 'UTC',
      tier = 'medium',
      seed,
      // ... other options
    } = options;

    this.id = id || `fp_${Date.now()}`;
    this.platformType = platform;  // MUST be assigned here
    this.timezone = timezone;
    // ... rest of initialization
  }
}
```

### Validation After Fix

```javascript
// Test with each platform
const platforms = ['windows', 'macos', 'linux'];

for (const platform of platforms) {
  const result = await client.sendCommand('create_fingerprint_profile', { 
    platform 
  });
  
  console.assert(
    result.profile.platformType === platform,
    `Platform mismatch: requested ${platform}, got ${result.profile.platformType}`
  );
  console.log(`✓ ${platform} profile: platformType correctly set`);
}
```

### Impact Assessment

**Low Impact Because:**
- User Agent strings appear to be generated correctly
- The platform parameter is being processed (correct User Agents are generated)
- Most detection systems check User Agent first, platform second
- Regional fingerprints are working correctly

**Should Still Fix Because:**
- Consistency is important for real deployments
- JavaScript-based detection might check `navigator.platform`
- Could cause false positives on strict detection services

4. **Regular regression testing** against detection services
