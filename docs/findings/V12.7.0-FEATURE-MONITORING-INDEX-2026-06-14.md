## FEATURE OVERVIEW

### What This Feature Does

**Real-time Performance Metrics & Monitoring System**

Provides comprehensive insight into Basset Hound Browser performance with:
- Live metrics on command latency, throughput, errors
- Session-level statistics
- Resource usage monitoring (memory, CPU, connections)
- Configurable alert thresholds
- 72-hour historical data retention
- WebSocket API for dashboard integration
- Real-time streaming (<100ms latency)

### Why It Matters

**Production Visibility:**
- Monitor broker/connector health
- Detect performance degradation early
- Track resource utilization
- Alert on anomalies automatically

**Integration Enablement:**
- Dashboard applications can display metrics in real-time
- External monitoring systems can query historical data
- Alert system triggers notifications
- Forensic analysis has performance context

## ARCHITECTURE AT A GLANCE

### 5-Module System

```
┌─────────────────────────────────────────┐
│  1. METRICS COLLECTOR (300-400 LOC)     │ ← Event source
│  Records: latency, throughput, errors   │
│  Emits: command, error, session events  │
└────────────────┬────────────────────────┘
                 ▼
┌─────────────────────────────────────────┐
│  2. METRICS AGGREGATOR (150-200 LOC)    │ ← Time-series processor
│  Aggregates: 1m, 5m, 1h windows         │
│  Calculates: trends, percentiles        │
└────────────────┬────────────────────────┘
                 ▼
┌─────────────────────────────────────────┐
│  3. METRICS STORE (100-150 LOC)         │ ← Historical data
│  Retains: 72-hour history with rings    │
│  Queries: time-range data access        │
└────────────────┬────────────────────────┘
                 ▼
         ┌───────┴────────┬──────────────┐
         ▼                ▼              ▼
    ┌────────┐      ┌──────────────┐ ┌────────┐
    │ Alert  │      │  WebSocket   │ │ Export │
    │Manager │      │   Commands   │ │(Prom)  │
    └────────┘      └──────────────┘ └────────┘
```

### Data Flow Pipeline

```
WebSocket Server
       │ command events
       ├─ session lifecycle events
       ├─ error events
       └─ connection lifecycle events
             ▼
    Metrics Collector (real-time)
             │ snapshots every 10-30s
             ▼
    Time-Series Store (historical)
             │ aggregations (1m, 5m, 1h)
             ├─────────────────┬────────────────┐
             ▼                 ▼                ▼
    API Query Handler    Alert Manager    Streaming Subscribers
```

## 10 WEBSOCKET COMMANDS IMPLEMENTED

```
1. get_metrics() 
   → Full current snapshot (all metrics)

2. get_performance_stats(timeRange)
   → Latency p50/p95/p99, throughput, success rate

3. get_session_stats(sessionId?)
   → Active/total sessions, avg duration, commands/session

4. get_resource_usage()
   → Memory, CPU, connection counts

5. get_performance_dashboard(timeRange)
   → All metrics + trends + alerts (dashboard-ready)

6. get_metric_history(startTime, endTime, granularity)
   → Historical data points (1m/5m/1h buckets)

7. stream_metrics(interval)
   → Real-time updates via WebSocket stream (push)

8. get_alerts(severity?, limit)
   → Active and recent alerts with thresholds

9. set_alert_threshold(alertType, value)
   → Update alert threshold (e.g., latency p99 = 150ms)

10. suppress_alert(alertType, durationMs)
    → Temporarily suppress alert type
```

## 8 ALERT TYPES & THRESHOLDS

| Alert Type | Default Threshold | When Triggered |
|------------|-------------------|----------------|
| HIGH_LATENCY | p99 > 100ms | Command latency spike |
| HIGH_ERROR_RATE | >5% errors | Too many failed commands |
| LOW_SUCCESS_RATE | <95% success | Insufficient command success |
| MEMORY_GROWTH | >10 MB/hour | Memory leak detected |
| CPU_OVERLOAD | >80% CPU | High processor utilization |
| CONNECTION_SPIKE | +50 in 1m | Sudden connection increase |
| SESSION_TIMEOUT | >1 hour | Long-running session |
| RESOURCE_EXHAUSTION | FDs near limit | System resource limits |

## CRITICAL IMPLEMENTATION NOTES

### 1. Use Ring Buffers for Historical Data
Avoid unbounded memory growth with fixed-size circular buffers:
```javascript
// Good: Fixed memory
this.oneMinute = new RingBuffer(1440);  // 24h of 1m data
this.oneHour = new RingBuffer(168);     // 7d of 1h data

// Bad: Unbounded
this.snapshots = [];  // Grows forever!
```

### 2. Use setImmediate() for Non-Blocking Metrics
Don't block command execution with metric recording:
```javascript
// Good: Non-blocking
recordCommand(name, duration, success) {
  setImmediate(() => {
    // Update metrics without blocking
  });
}

// Bad: Blocking
recordCommand(name, duration, success) {
  // Updates metrics synchronously
}
```

### 3. Sort Once, Reuse for Percentiles
Calculate percentiles from already-sorted data:
```javascript
// Good: O(n) sort once
const sorted = [...this.samples].sort((a,b) => a-b);
this.p50 = sorted[Math.floor(0.50 * sorted.length)];
this.p95 = sorted[Math.floor(0.95 * sorted.length)];
this.p99 = sorted[Math.floor(0.99 * sorted.length)];

// Bad: O(n log n) sort for each percentile
this.p50 = percentile(this.samples, 0.50);
this.p95 = percentile(this.samples, 0.95);
this.p99 = percentile(this.samples, 0.99);
```

### 4. Alert Suppression to Prevent Spam
Track suppression with auto-expiry:
```javascript
// Suppress 'high_latency' alerts for 5 minutes
suppressAlert('high_latency', 5 * 60 * 1000);

// Check before alerting:
if (!this._isSuppressed('high_latency')) {
  // Generate alert
}

// Auto-cleanup in periodic tick:
for (const [type, until] of this.suppressions) {
  if (Date.now() > until) this.suppressions.delete(type);
}
```

### 5. Time-Window Boundaries
Use floor() for consistent bucketing across all events:
```javascript
function getBucket(timestamp, windowMs) {
  return Math.floor(timestamp / windowMs) * windowMs;
}
// All events in [10:00:00, 10:00:59] → 10:00:00 bucket
// Consistent across all aggregations
```
