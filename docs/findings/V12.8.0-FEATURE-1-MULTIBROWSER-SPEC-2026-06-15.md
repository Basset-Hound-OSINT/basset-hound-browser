## EXECUTIVE SUMMARY

### The Problem
Currently, Basset Hound Browser can only control browser instances via **Electron-based deployment**. To expand to different use cases (Docker-light environments, CI/CD systems, cloud deployments), we need support for **Chrome and Firefox** while maintaining the **same WebSocket API**. Users should be able to switch browsers without code changes.

### The Solution
Implement a **browser abstraction layer** that:
1. Defines unified driver interface
2. Implements Chrome Remote Debugging Protocol (CDP) driver
3. Implements Firefox WebDriver protocol driver
4. Routes all commands to appropriate driver
5. Handles browser-specific capability differences

### The Benefit
- **Ecosystem expansion:** Support Chrome, Firefox, Safari (future)
- **Flexibility:** Users choose their browser engine
- **Simplicity:** Single API across all browsers
- **Compatibility:** Existing Electron code unchanged
- **Scalability:** Pool manager can manage mixed browser fleets

## ARCHITECTURE OVERVIEW

### Layer Model

```
WebSocket API (v12.8.0)
├── 170+ Existing Commands (unchanged)
├── 6 New Commands (browser management)
└── Modified: All commands accept optional `browser_id` parameter

Browser Abstraction Layer (new)
├── Driver Interface (abstract)
├── Chrome Driver (CDP)
├── Firefox Driver (WebDriver)
├── Electron Driver (refactored from existing)
└── Auto-selection logic

Backend Drivers
├── Chrome/Chromium (via CDP WebSocket)
├── Firefox (via WebDriver HTTP/JSON-RPC)
└── Electron (via Node.js IPC)
```

### Design Principles

1. **Unified Interface:** All drivers implement same method signatures
2. **Browser Agnostic:** Higher layers don't know browser type
3. **Feature Parity:** All commands work on all browsers (where applicable)
4. **Transparent Routing:** Automatic driver selection
5. **No Breaking Changes:** Electron deployment unaffected

#### Browser Pool (Instance Management)

```javascript
// src/managers/browser-pool.js (400-600 LOC)
class BrowserPool {
  constructor(config = {}) {
    this.drivers = {}; // {id: driver instance}
    this.config = config;
  }
  
  async createBrowser(browserType = 'electron', options = {}) {
    const driver = await BrowserFactory.createDriver(browserType, options);
    await driver.initialize();
    const id = this.generateId();
    this.drivers[id] = {
      type: browserType,
      driver,
      createdAt: Date.now(),
      lastUsed: Date.now()
    };
    return { id, type: browserType };
  }
  
  async getBrowser(id) {
    if (!this.drivers[id]) throw new Error(`Browser ${id} not found`);
    return this.drivers[id].driver;
  }
  
  async closeBrowser(id) {
    if (!this.drivers[id]) throw new Error(`Browser ${id} not found`);
    await this.drivers[id].driver.close();
    delete this.drivers[id];
  }
  
  async listBrowsers() {
    return Object.entries(this.drivers).map(([id, data]) => ({
      id,
      type: data.type,
      createdAt: data.createdAt,
      lastUsed: data.lastUsed
    }));
  }
  
  async closeAll() {
    for (const [id, data] of Object.entries(this.drivers)) {
      await data.driver.close();
    }
    this.drivers = {};
  }
}
```

## ERROR HANDLING & EDGE CASES

### Browser Not Found
- If `browser_id` specified but doesn't exist: return 404 error
- Suggest available browsers in error message

### Browser Crash
- Detect via health check (ping interval)
- Auto-restart for critical browsers (configurable)
- Notify client of crash with error code

### Connection Loss
- Attempt reconnect with exponential backoff
- Timeout after 30 seconds, mark as dead
- Clean up resources and notify client

### Feature Not Supported
- Check capability matrix before executing
- Return error with list of supported browsers
- Suggest alternative approaches if available

### Timeout
- Navigation timeout: 30 seconds default (configurable)
- Element wait timeout: 10 seconds default (configurable)
- Network timeout: 60 seconds default (configurable)
