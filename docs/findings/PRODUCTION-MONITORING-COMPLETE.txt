ALERT RULES REFERENCE
================================================================================

CRITICAL RULES (4)
──────────────────
1. high_command_failure_rate
   ├─ Threshold: >50 failed commands/minute
   ├─ Impact: Commands not executing
   └─ Action: Check WebSocket server

2. no_active_connections
   ├─ Threshold: 0 active connections
   ├─ Impact: No client connections
   └─ Action: Server health check

3. disk_space_critical
   ├─ Threshold: >90% disk usage
   ├─ Impact: Out-of-space risk
   └─ Action: Clean up or expand

4. excessive_reconnections
   ├─ Threshold: >10 in 5 minutes
   ├─ Impact: Connection instability
   └─ Action: Network diagnosis

HIGH PRIORITY RULES (8)
──────────────────────
- high_message_latency (>1000ms)
- high_memory_usage (>500MB heap)
- memory_pressure (>85% system)
- high_cpu_usage (>80%)
- high_error_rate (sudden increase)
- message_broker_error (error spike)
- large_message_queue (>1000 messages)
- high_page_load_time (>30 seconds)

KNOWN LIMITATIONS & CONSIDERATIONS
================================================================================

1. METRICS RETENTION
   ├─ Default: 1 hour
   ├─ Impact: Older metrics not available
   └─ Solution: Increase retentionPeriod or use Prometheus for long-term

2. ALERT COOLDOWN
   ├─ Default: 5 minutes
   ├─ Impact: Same alert won't trigger within 5 minutes
   └─ Solution: Adjust cooldownDuration in rule config

3. HEALTH CHECK TIMEOUT
   ├─ Default: 5 seconds
   ├─ Impact: Checks exceeding timeout marked as DOWN
   └─ Solution: Increase checkTimeout for slow services

4. INCIDENT AUTO-CREATION
   ├─ Only for high/critical alerts
   ├─ Medium/low alerts don't auto-create
   └─ Solution: Create manual incidents as needed

5. STORAGE
   ├─ Incident data stored in JSON files
   ├─ No built-in data cleanup
   └─ Solution: Implement periodic archival
