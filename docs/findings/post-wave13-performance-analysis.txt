## SECTION 2: OPTIMIZATION OPPORTUNITIES (REMAINING)

### OPT-14: Connection Pool Per-Domain Optimization (Wave 15)

**Concept:** Separate connection pools per domain to prevent head-of-line blocking

**Current State:**
- Single global connection pool (48 connections)
- All domains share same queue
- High-latency domains can block fast domains

**Proposed Solution:**
- 1 primary pool (same 48) for general traffic
- 5-10 domain-specific mini-pools (8-16 connections each)
- Auto-create pool for domains with >10 pending requests
- Auto-evict pool when idle >5 minutes

**Implementation:**
```javascript
// Create pool per domain
const domainPools = new Map();
function getOrCreatePool(domain) {
  if (!domainPools.has(domain)) {
    domainPools.set(domain, new ConnectionPool({
      size: 12,
      domain,
      priority: domain === 'blocked-domain' ? 'low' : 'normal'
    }));
  }
  return domainPools.get(domain);
}

// Route request to appropriate pool
function routeRequest(domain, request) {
  const pool = getOrCreatePool(domain);
  pool.enqueue(request);
}
```

**Effort:** 20-30 hours
**Impact:** +5-10% throughput (reduces blocking), no latency gain
**Risk:** LOW (graceful fallback to global pool)
**Timeline:** Wave 15 (after Wave 13 Phase 2 complete)

### OPT-15: Streaming Screenshot Response (Wave 15)

**Concept:** Stream large screenshots to client instead of buffering in memory

**Current State:**
- Screenshot encoded entirely in memory
- 100-500KB images held in buffer
- Causes memory spikes under load
- Blocks next screenshot until encoding complete

**Proposed Solution:**
- Chunked encoding (split large images into 64KB chunks)
- Stream chunks to client as available
- Client reassembles in WebSocket handler
- Reduces peak memory by 60-80%

**Implementation:**
```javascript
async function streamScreenshot(dimensions) {
  const encoder = new WebPStreamer({ chunkSize: 65536 });

  // Encode in chunks, stream immediately
  encoder.on('chunk', (buffer, offset) => {
    this.client.send(JSON.stringify({
      type: 'screenshot_chunk',
      offset,
      size: buffer.length,
      data: buffer.toString('base64')
    }));
  });

  encoder.on('complete', (metadata) => {
    this.client.send(JSON.stringify({
      type: 'screenshot_complete',
      hash: metadata.hash,
      size: metadata.totalSize
    }));
  });

  await encoder.encode(canvas);
}
```

**Effort:** 30-40 hours
**Impact:** -60-80% screenshot memory spike, enables larger images
**Risk:** MEDIUM (streaming state tracking required)
**Timeline:** Wave 15-16 (post v12.2.0)

### OPT-16: Request Batching & Pipelining (Wave 15)

**Concept:** Allow client to pipeline multiple commands, server processes in batch

**Current State:**
- One command at a time (request/response)
- Client must wait for response before sending next
- ~10-20ms round-trip latency per command
- Serial processing limits throughput for multi-step workflows

**Proposed Solution:**
- Client sends array of commands: `[cmd1, cmd2, cmd3]`
- Server processes in parallel/sequential (configurable)
- Returns array of results: `[result1, result2, result3]`
- Reduces round-trips by 70%

**Implementation:**
```javascript
// Client side
async function batchNavigateAndExtract(urls) {
  return await ws.request({
    batch: true,
    commands: [
      { cmd: 'navigate', url: urls[0] },
      { cmd: 'wait_for_element', selector: 'body' },
      { cmd: 'get_content' },
      { cmd: 'navigate', url: urls[1] },
      { cmd: 'wait_for_element', selector: 'body' },
      { cmd: 'get_content' }
    ]
  });
}

// Server side
async handleBatch(commands) {
  const results = [];
  for (const cmd of commands) {
    const result = await this.commandHandlers[cmd.cmd](cmd);
    results.push(result);
  }
  return { batch_results: results };
}
```

**Effort:** 25-35 hours
**Impact:** +20-30% throughput for multi-step workflows, -70% round-trip latency
**Risk:** LOW (graceful fallback to single commands)
**Timeline:** Wave 15 (after Wave 13 Phase 2)

### OPT-17: Fingerprint Profile Lazy Generation (Wave 15)

**Concept:** Generate fingerprint profiles on-demand rather than pre-loading all

**Current State:**
- All 8 fingerprint profiles loaded at startup
- Takes 200-400ms startup time
- Memory used even if profiles never accessed
- Startup delay impacts rapid-fire deployments

**Proposed Solution:**
- Profiles generated on first use
- Cache generated profiles (LRU, max 50)
- Background refresh for frequently used profiles
- Reduces startup time by 50%

**Implementation:**
```javascript
class LazyFingerprintManager {
  constructor() {
    this.cache = new LRU({ max: 50 });
    this.generating = new Map(); // Track in-flight generations
  }

  async getProfile(platform, risk) {
    const key = `${platform}-${risk}`;

    // Return from cache if available
    if (this.cache.has(key)) {
      return this.cache.get(key);
    }

    // Wait if generation in-flight
    if (this.generating.has(key)) {
      return await this.generating.get(key);
    }

    // Generate on-demand
    const promise = generateProfile(platform, risk);
    this.generating.set(key, promise);

    try {
      const profile = await promise;
      this.cache.set(key, profile);
      return profile;
    } finally {
      this.generating.delete(key);
    }
  }
}
```

**Effort:** 15-20 hours
**Impact:** -50% startup time (200-400ms → 100-200ms), -5MB memory baseline
**Risk:** LOW (cache fallback works fine)
**Timeline:** Wave 15 (quick win)

### OPT-18: Behavioral AI Path Precompilation (Wave 15)

**Concept:** Precompile behavioral AI paths (mouse, typing) instead of computing per-request

**Current State:**
- Mouse paths computed on each move command: 20-50ms
- Typing patterns computed per key: 5-10ms
- Complex physics calculations for each interaction
- CPU-intensive for high-concurrency scenarios

**Proposed Solution:**
- Precompile common paths (straight, curved, zigzag, etc.)
- Cache behavioral patterns by session type
- Lookup from cache instead of recomputing
- Reduces per-interaction CPU by 60-80%

**Implementation:**
```javascript
class BehavioralAICache {
  constructor() {
    this.pathCache = new Map();
    this.typingCache = new Map();
    this.precompile();
  }

  precompile() {
    // Pre-generate common paths
    const paths = [];
    for (let distance = 10; distance <= 500; distance += 50) {
      for (const style of ['linear', 'curved', 'jittered']) {
        const path = computePath(distance, style);
        this.pathCache.set(`${distance}-${style}`, path);
      }
    }

    // Pre-generate common typing patterns
    for (const speed of ['slow', 'normal', 'fast']) {
      const pattern = computeTypingPattern(speed);
      this.typingCache.set(speed, pattern);
    }
  }

  getPath(distance, style = 'curved') {
    return this.pathCache.get(`${distance}-${style}`) ||
           computePath(distance, style); // Fallback
  }
}
```

**Effort:** 20-25 hours
**Impact:** -60-80% behavioral AI CPU, +10-15% throughput under high concurrency
**Risk:** LOW (precompiled patterns are deterministic)
**Timeline:** Wave 15 (quick win)
