PART 2: PERFORMANCE BOTTLENECK ANALYSIS
================================================================================

Bottleneck Distribution (Where Time Is Spent):

1. WebSocket Message Processing (40% of latency)
   - Frame parsing: 50-100µs
   - Command routing: 30-80µs
   - Parameter validation: 20-50µs
   - Response serialization: 40-100µs
   - Total: 140-330µs per message

   Scaling issue: O(n) command lookup (linear search in 164 commands)

   Optimization potential: Hash-based routing (-50-70%)
   Estimated gain: 70-150µs/message (20-25% throughput improvement)

2. Browser Interaction (35% of latency)
   - DOM query (CSS selector): 10-50ms
   - Screenshot capture: 50-200ms
   - JavaScript execution: 5-30ms
   - Navigation/page load: 500-5000ms

   Scaling issue: JS execution is single-threaded per BrowserView

   Optimization potential: Parallel screenshot processing, DOM caching
   Estimated gain: 30-50ms/operation (15-20% throughput improvement)

3. Disk I/O (15% of latency)
   - Screenshot write: 10-50ms
   - Checkpoint save: 5-20ms
   - Log write: 1-5ms
   - HAR capture: 5-15ms

   Scaling issue: Synchronous writes on same disk

   Optimization potential: Async writes, compression, batching
   Estimated gain: 8-25ms/operation (10-15% throughput improvement)

4. Network I/O (10% of latency)
   - Tor exit lookup: 50-100ms
   - Proxy reputation check: 20-50ms
   - External API calls: 100-500ms
   - DNS resolution: 10-50ms

   Scaling issue: Blocking calls to external services

   Optimization potential: Caching, async calls, connection pooling
   Estimated gain: 30-100ms/operation (5-10% throughput improvement)

Concurrency Bottleneck Analysis:

Single-Browser Constraints:
- One BrowserView per session: Max 1 navigation at a time
- JavaScript execution: Single-threaded per context
- Network: Limited by OS file descriptors

Multi-Client Constraints:
- WebSocket server: Node.js event loop (single-threaded)
- Connection limit: OS-dependent (typically 10K per process)
- Memory per client: ~2.5 MB baseline, higher with active operations

System Limits:
- OS file descriptor limit: ~1024 per process (typically)
- TCP connection backlog: 128 (configurable)
- Memory: 45 GB available (not a limit at <1% utilization)
- CPU: 8 cores (not a limit at <25% utilization)

PART 3: PERFORMANCE IMPROVEMENT ROADMAP
================================================================================

QUICK WINS (Phase 2, 20-40 hours, +20-25% throughput)

1. Hash-Based Command Routing (8 hours, +100µs/message)
   Current: Linear search through 164 commands
   Improvement: Hash map lookup O(1)
   Impact: +0.8-1.2 msg/sec per 10 concurrent clients
   Expected gain: +50-75 msg/sec at 200 concurrent (17% improvement)
   Implementation:
   - Create command router hash map in server.js
   - Update command registration to populate hash
   - Replace switch statement with hash lookup

2. DOM Query Caching (12 hours, -15-30ms)
   Current: Re-query same CSS selectors on each operation
   Improvement: Cache selector results for 5-10 seconds
   Impact: +15-20% throughput for multi-selector operations
   Expected gain: +30-50 msg/sec (10% improvement)
   Implementation:
   - Add query cache to content extractor
   - Implement 5-second TTL with invalidation
   - Cache hit rate monitoring

3. Async Screenshot Writing (10 hours, -20-30ms)
   Current: Synchronous disk write blocks processing
   Improvement: Queue screenshots for async write
   Impact: +30-40% throughput for screenshot-heavy workloads
   Expected gain: +40-60 msg/sec (15% improvement)
   Implementation:
   - Create screenshot write queue
   - Implement async write batching
   - Add backpressure handling

4. Connection Pooling (8 hours, -20-40ms)
   Current: New socket per request for external APIs
   Improvement: Reuse connections for Tor, proxy APIs
   Impact: +5-10% throughput for proxy-heavy workloads
   Expected gain: +20-30 msg/sec (5% improvement)
   Implementation:
   - Add HTTP agent pooling
   - Reuse Tor SOCKS connections
   - Implement connection timeout/cleanup

MEDIUM-TERM IMPROVEMENTS (Phase 3, 40-80 hours, +30-40% throughput)

5. Parallel Screenshot Processing (15 hours, -50-100ms)
   Current: One screenshot at a time per browser
   Improvement: Capture multiple screenshots in parallel across browser instances
   Impact: +40-60% throughput for screenshot workloads
   Expected gain: +80-120 msg/sec (25% improvement)
   Implementation:
   - Create screenshot worker pool
   - Distribute captures across workers
   - Implement rate limiting per worker

6. Multi-Process WebSocket Server (20 hours, +2-3x)
   Current: Single Node.js process (one event loop)
   Improvement: Run 4-8 processes behind load balancer
   Impact: 4-8x throughput potential (limited by bottleneck #2)
   Expected gain: +3-4x throughput (300% improvement, but limited by browser)
   Implementation:
   - Create cluster setup with primary + workers
   - Use IPC for state sharing
   - Implement session affinity in load balancer

7. Smart DOM Diff (25 hours, -20-40ms)
   Current: Full DOM tree comparison
   Improvement: Only diff changed regions
   Impact: +20-30% throughput for page monitoring
   Expected gain: +50-80 msg/sec (20% improvement)
   Implementation:
   - Implement incremental DOM diffing
   - Cache DOM structure
   - Region-based change detection

8. JavaScript Execution Sandbox Optimization (15 hours, -10-20ms)
   Current: Full sandbox overhead per execution
   Improvement: Reuse sandbox contexts
   Impact: +15-25% throughput for JS execution
   Expected gain: +30-50 msg/sec (15% improvement)
   Implementation:
   - Pool sandbox contexts
   - Cache compiled scripts
   - Implement context cleanup

LONG-TERM STRATEGIC IMPROVEMENTS (Phase 4, 80-160 hours, +100-150% throughput)

9. Distributed Browser Instances (40 hours, +3-5x)
   Current: Single browser instance per session
   Improvement: Distribute pages across browser pool
   Impact: +3-5x throughput for page-heavy workloads
   Expected gain: +600-1200 msg/sec (200-400% improvement)
   Implementation:
   - Create browser pool manager
   - Distribute page operations across pool
   - Implement session-to-browser affinity

10. Native C++ Bridge (60 hours, -20-50% latency)
    Current: Node.js JS code for all operations
    Improvement: Move hot paths to C++ (screenshot, DOM diff)
    Impact: +30-50% throughput
    Expected gain: +100-200 msg/sec (30-50% improvement)
    Implementation:
    - Profile hottest operations
    - Implement C++ bindings (Node.js N-API)
    - Replace JS implementations for screenshot/DOM ops

11. Database-Backed State Management (40 hours, +2-3x)
    Current: In-memory state (limited scaling)
    Improvement: Use TimescaleDB for state persistence
    Impact: +2-3x scaling capacity
    Expected gain: Support 500+ concurrent clients
    Implementation:
    - Add TimescaleDB integration
    - Implement state serialization
    - Add connection pooling

PART 5: CONCURRENT CLIENT SCALING ANALYSIS
================================================================================

Current Limit: 200 stable, 300 degraded (>3ms latency)

Bottleneck: WebSocket message processing (40%) + browser interaction (35%)

Scaling Path to 500 Concurrent Clients:

Strategy 1: Vertical Scaling (Upgrade Hardware)
- Current: 8 cores, 45 GB RAM
- Upgraded: 16 cores, 128 GB RAM
- Expected improvement: +50-75% throughput (more cores, more connections)
- Realistic ceiling: 300-350 concurrent clients
- Cost: $10-20K/month (high-end cloud instance)
- ROI: Limited (browser interaction is bottleneck)

Strategy 2: Horizontal Scaling (Multiple Processes)
- Current: Single Node.js process
- Upgraded: 4-8 processes behind load balancer
- Expected improvement: 3-5x throughput (parallelization)
- Realistic ceiling: 500-800 concurrent clients
- Cost: $30-50K/month (4-8 instances)
- ROI: High (scales linearly with load)

Strategy 3: Distributed Browser Instances
- Current: Single browser per session
- Upgraded: Browser pool with 4-8 instances
- Expected improvement: 3-5x throughput (page operations parallel)
- Realistic ceiling: 800-1200 concurrent clients
- Cost: $60-100K/month (16-32 instances)
- ROI: Highest (true parallelization)

Strategy 4: Hybrid (Process + Browser Scaling)
- 4 Node.js processes + 2 browser pools (4-8 instances each)
- Expected improvement: 8-16x throughput
- Realistic ceiling: 1600-3200 concurrent clients
- Cost: $100-150K/month
- ROI: Highest, but overkill for current market demand

Recommended Path for Wave 15:
1. Phase 2 (Jun-Aug): Quick wins only (+20-25%, cost ~$0)
2. Phase 3 (Sep-Nov): Medium-term improvements + horizontal scaling (+50-80%, cost ~$30K/month)
3. Phase 4 (Dec 2026+): Long-term if market demand justifies ($60K+/month)
