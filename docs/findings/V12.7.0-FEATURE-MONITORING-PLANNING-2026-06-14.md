## ARCHITECTURE OVERVIEW

### System Design

```
┌─────────────────────────────────────────────────────────┐
│                  WebSocket Client                       │
│         (Dashboard, Monitoring Service, API)            │
└────────────────────┬────────────────────────────────────┘
                     │ JSON Commands
                     ▼
┌─────────────────────────────────────────────────────────┐
│            WebSocket Command Handler                    │
│  (get_metrics, get_performance_stats, stream_metrics)   │
└──────────┬──────────────────────────────────────────────┘
           │
     ┌─────┴──────────┬──────────────┬─────────────────┐
     ▼                ▼              ▼                 ▼
┌──────────┐   ┌──────────────┐ ┌────────┐   ┌──────────────┐
│ Metrics  │   │   Metrics    │ │ Alert  │   │  Historical  │
│Collector │   │ Aggregator   │ │Manager │   │    Store     │
└──────────┘   └──────────────┘ └────────┘   └──────────────┘
     │                │              │              │
     ├─ Command Events │ Time-Series  │ Threshold   │ Snapshots
     ├─ Session Events │ Aggregation  │ Violations  │ Trends
     ├─ Error Events   │ Percentiles  │ Alerts      │ Analysis
     └─ Connection Events
```

### Data Flow

```
1. Event Source (WebSocket Server)
   ├─ Command execution events
   ├─ Session lifecycle events
   ├─ Error events
   └─ Connection lifecycle events
   
2. Metrics Collector
   ├─ Record latency samples
   ├─ Update counters
   ├─ Track resource usage
   └─ Calculate throughput
   
3. Time-Series Store
   ├─ 1-minute aggregations
   ├─ 5-minute aggregations
   ├─ 1-hour aggregations
   └─ Historical trend data
   
4. Alert System
   ├─ Check threshold violations
   ├─ Generate alerts
   ├─ Store alert history
   └─ Deduplicate/suppress alerts
   
5. Query API
   ├─ Serve current metrics
   ├─ Return historical data
   ├─ Stream real-time updates
   └─ Query alert status
```

  // Send confirmation
  sendSuccess(ws, message.id, { streamId });
};

```javascript
// At server startup
const metricsCollector = new MetricsCollector();
const metricsAggregator = new MetricsAggregator();
const alertManager = new AlertManager();
const metricsStore = new MetricsStore();

// Hook into command execution
const origDispatch = commandDispatcher.dispatch;
commandDispatcher.dispatch = function(command, params) {
  const startTime = Date.now();
  const startId = generateId();
  
  // Record start
  metricsCollector.recordCommandStart(command, startId);
  
  // Execute
  const result = origDispatch.call(this, command, params);
  
  // Record completion
  const duration = Date.now() - startTime;
  metricsCollector.recordCommandEnd(
    startId,
    command,
    duration,
    result.success,
    result.dataSize || 0
  );
  
  return result;
};

// Hook into session creation
origSessionCreate.on = function(sessionId) {
  metricsCollector.recordSessionCreated(sessionId);
  return origSessionCreate.apply(this, arguments);
};

// Hook into error handling
errorHandler.on('error', (error, command) => {
  metricsCollector.recordError(
    error.type,
    error.message,
    command,
    error.stack
  );
});
```

## RISK MITIGATION

### Risk 1: Metrics Collection Overhead
**Risk:** Metric collection adds significant latency to commands
**Probability:** Medium (40%)
**Mitigation:**
- Use async event emission where possible
- Batch percentile calculations
- Limit sample buffer to 10k entries
- Profile implementation early

### Risk 2: Memory Growth Over 72 Hours
**Risk:** Historical data store consumes unbounded memory
**Probability:** Medium (35%)
**Mitigation:**
- Use ring buffers for fixed-size storage
- Automatic cleanup of old snapshots
- Compression of historical data
- Monitor memory growth rate

### Risk 3: Streaming Latency Under Load
**Risk:** Real-time streaming degrades with many subscribers
**Probability:** Low-Medium (25%)
**Mitigation:**
- Implement efficient event broadcasting
- Limit concurrent streams
- Optional metric sampling (send every Nth sample)
- Alert if streaming latency exceeds threshold

### Risk 4: Alert Spam
**Risk:** Alerts trigger repeatedly for same condition
**Probability:** Medium (40%)
**Mitigation:**
- Implement alert suppression/deduplication
- Configurable minimum duration between alerts
- Alert state tracking (new/ongoing/resolved)
