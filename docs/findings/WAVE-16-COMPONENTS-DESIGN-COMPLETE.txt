KEY DESIGN DECISIONS
================================================================================

1. LOAD BALANCING: HAProxy over ALB
   Rationale: Better WebSocket support, connection pooling, lower latency

2. SESSION STORE: Redis Sentinel (separate from cache)
   Rationale: Dedicated session tier ensures consistency, clear separation

3. STREAM PROCESSING: Kafka Streams over Lambda
   Rationale: Deterministic latency (<100ms), full control, cost-effective

4. DATABASE: PostgreSQL RDS over DynamoDB
   Rationale: Complex schema, transactions, analytics queries required

5. CACHE: Redis (separate from session store)
   Rationale: Dedicated caching tier for configuration, change history

6. AUTHENTICATION: OAuth2 + API keys + mTLS
   Rationale: Supports multiple use cases (web, programmatic, service-to-service)

7. OBSERVABILITY: 3-pillar approach (metrics + logs + traces)
   Rationale: Complete visibility, debugging, performance optimization

8. MULTI-REGION: Active-passive (US primary, EU standby)
   Rationale: GDPR compliance, disaster recovery, cost optimization

RISKS & MITIGATIONS
================================================================================

HIGH-RISK AREAS IDENTIFIED:

1. Database Bottleneck
   Risk: Write throughput maxed at 50k/sec
   Mitigation: Partitioning strategy, read replicas, caching layer

2. Redis Failover Complexity
   Risk: Sentinel misconfiguration → data loss
   Mitigation: Strict configuration management, extensive testing, backup layer

3. Kubernetes Complexity
   Risk: Unfamiliar operational patterns
   Mitigation: Training, automation, runbooks, SRE support

4. Cross-Region Replication
   Risk: High latency (>100ms) → consistency issues
   Mitigation: Eventual consistency model, async replication, conflict resolution

5. Cost Overruns
   Risk: Actual costs exceed estimates (88k/month)
   Mitigation: Auto-scaling limits, reserved instances, data tiering

All risks documented in component designs with specific mitigations.
