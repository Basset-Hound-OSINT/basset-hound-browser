PART 1: PROFILING ANALYSIS - BOTTLENECK IDENTIFICATION
================================================================================

PERFORMANCE CEILING ANALYSIS:

Bottleneck Distribution (Where latency is spent):
- WebSocket message processing: 40% (114µs total)
  - Frame parsing: 50-100µs
  - Command routing: 30-80µs ← PRIMARY BOTTLENECK
  - Parameter validation: 20-50µs
  - Response serialization: 40-100µs

- Browser interaction: 35% (10-50ms per operation)
  - DOM query (CSS selector): 10-50ms
  - Screenshot capture: 50-200ms
  - JavaScript execution: 5-30ms ← SECONDARY BOTTLENECK (context creation)
  - Navigation: 500-5000ms (not bottleneck for throughput)

- Disk I/O: 15% (10-50ms per write)
  - Screenshot writes: 10-50ms ← TERTIARY BOTTLENECK
  - Checkpoint saves: 5-20ms
  - Log writes: 1-5ms

- Network I/O: 10% (50-100ms for external calls)
  - Tor exit lookups: 50-100ms ← CACHE OPPORTUNITY
  - Proxy reputation checks: 20-50ms ← CACHE OPPORTUNITY
  - DNS resolution: 10-50ms (cached by OS)
  - External APIs: 100-500ms (rare)

ROOT CAUSE ANALYSIS:

1. LINEAR COMMAND ROUTING (40% of message processing)
   File: /websocket/server.js, handleCommand() method
   Current: if/else chain with 164 commands
   Issue: Each message requires ~80-90 string comparisons on average
   Cost: 30-80µs per message (at 200 concurrent = 40% latency)
   Impact: Single biggest bottleneck in message processing

   Analysis:
   - Current implementation uses switch statement (compiles to linear search)
   - With 164 commands, average case = 82 comparisons
   - Worst case (last command) = 164 comparisons
   - Best case (first command) = 1 comparison
   - Cost per comparison: ~0.5-1µs (string equality)
   - Average total: 30-80µs per message

2. DOM EXTRACTION RE-PARSING (20-30ms per operation)
   File: /inspector/manager.js, DOMInspector class
   Current: Extract HTML, then text, then links = 3x parsing
   Issue: Each extraction re-parses entire DOM tree
   Cost: 20-30ms per operation
   Pattern: Typical workflow (get_html + get_text + get_links) = 60-90ms
   Impact: 25-50% of browser interaction latency

   Analysis:
   - DOM traversal: 10-20ms (Chromium engine overhead)
   - Selector parsing: 2-5ms
   - Serialization: 5-10ms
   - Total: 20-30ms per operation
   - 3 sequential operations: 60-90ms (cache could reduce to 25-35ms)

3. SYNCHRONOUS SCREENSHOT DISK I/O (10-50ms blocking)
   File: /screenshots/manager.js, screenshot write operations
   Current: Capture → Encode → WRITE (blocks response)
   Issue: Disk write blocks entire request processing
   Cost: 10-50ms per screenshot (synchronous)
   Impact: 15% of disk I/O latency

   Analysis:
   - Screenshot capture: 50-200ms (browser operation)
   - Encoding: 10-50ms (compression)
   - Disk write: 10-50ms (SYNCHRONOUS - BLOCKING)
   - Total: 70-300ms (write blocks processing)
   - Optimization: Return immediately, queue write for background flush

4. JAVASCRIPT CONTEXT CREATION (5-10ms overhead)
   File: Browser execution context creation
   Current: New context per execute_script call
   Issue: Context creation is expensive (sandbox setup)
   Cost: 5-10ms per execution
   Impact: 15-25% of JS execution latency

   Analysis:
   - Context creation: 5-10ms (sandbox + environment setup)
   - Script execution: 1-5ms (actual code)
   - Cleanup: 1-2ms
   - Reuse strategy: Pool context, reuse across calls
   - Benefit: Reduce context creation overhead by 90%

5. EXTERNAL API CALLS (50-100ms, high miss cost)
   File: /proxy/manager.js, /evasion/* files
   Current: Real-time lookups for Tor nodes, proxy reputation
   Issue: Blocking external API calls without caching
   Cost: 50-100ms per lookup (uncached)
   Hit rate: 80-90% (same queries repeated)
   Impact: 10% of total latency, but easily cached

   Analysis:
   - Tor node lookup: 50-100ms (external service)
   - Proxy reputation: 20-50ms (external service)
   - Cache potential: 80-90% hit rate (typical OSINT patterns)
   - Cache benefit: 50-100ms → <1ms on hit (99%+ improvement)
   - TTL: 1 hour (Tor) to 24 hours (user agents) acceptable

- Integration: /screenshots/manager.js
- Methods: queueScreenshot(), flush()

Queue Management:
- Enqueue immediately on capture
- Flush automatically every 100ms
- Flush immediately if queue > 50 items
- Persist queue on shutdown
- Recover on restart

Testing:

- Pool size: 8-16 pre-created contexts
- Context cleanup: On error or timeout
- Timeout: 30 seconds per context (kill if hung)
- Memory per context: ~500KB (total ~5-8 MB)

Impact Calculation:
- Context creation overhead: 5-10ms per execution
- Pool reuse: Eliminates creation on most calls
- Throughput improvement: +15% for JS-heavy workloads

Implementation Location:
- File: Create /sandbox/context-pool.js
- Integration: /extraction/* and /browser/* files
- Methods: acquireContext(), releaseContext()

Pool Management:
- Acquire context from pool (returns immediately if available)
- Execute script in context
- Release context back to pool (clean state)
- Monitor: Pool utilization, context failures
- Shutdown: Cleanup all contexts gracefully

Testing:
- Verify contexts are reused (not recreated)
