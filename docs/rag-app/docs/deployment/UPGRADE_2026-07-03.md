# Upgrading a pre-2026-07-03 copy of rag-bootstrap

**Audience**: developers/teams who copied `rag-bootstrap/` into their project
**before 2026-07-03** and want to pull the 2026-07-03 overhaul from this
canonical repo.
**Verification**: the upgraded stack passed a full live smoke on 2026-07-03
(doctor 8/8, up in 12s, explicit host-path ingest + idempotent re-ingest,
search, teardown — no-arg `./deploy.sh ingest` was not exercised; see step 4's
`ingestion.directories` alignment) — see
[findings/audits/2026-07-03_live_smoke_report.md](../findings/audits/2026-07-03_live_smoke_report.md).

The overhaul changes **defaults and one API contract**. Read the two breaking
changes first; everything else is additive.

---

## Breaking change 1 — port scheme: 8100 → 10000 (`RAG_PORT_BASE`)

The old default web port was **8100**. All published host ports now derive from
a single base, `RAG_PORT_BASE=10000`. `deploy.sh` computes
`RAG_<svc>_PORT = RAG_PORT_BASE + offset` and writes the concrete values into
`.env`; the compose files consume `${RAG_<svc>_PORT:-<default>}`.
Container-side ports are unchanged.

| Service | Env var | Offset | Default host port | Container port |
|---------|---------|--------|-------------------|----------------|
| web (nginx) | `RAG_PORT` | +0 | **10000** | 80 |
| prometheus | `RAG_PROMETHEUS_PORT` | +10 | 10010 | 9090 |
| grafana | `RAG_GRAFANA_PORT` | +11 | 10011 | 3000 |
| loki | `RAG_LOKI_PORT` | +12 | 10012 | 3100 |
| alertmanager | `RAG_ALERTMANAGER_PORT` | +13 | 10013 | 9093 |
| postgres-exporter | `RAG_PG_EXPORTER_PORT` | +14 | 10014 | 9187 |
| redis-exporter | `RAG_REDIS_EXPORTER_PORT` | +15 | 10015 | 9121 |
| cadvisor | `RAG_CADVISOR_PORT` | +16 | 10016 | 8080 |

(Rows +10..+16 belong to the monitoring overlay, which is **out of scope /
unsupported for now** — see the README monitoring note.)

- Reserved band for one stack: **10000–10019**; second instances and
  downstream apps take **10020 and up**.
- Ollama stays on its well-known external port `11434` (not remapped).
- Internal-only ports (postgres 5432, redis 6379, api 8000) are never published.
- If `RAG_PORT` is busy at start, `deploy.sh` auto-selects the next free port
  in the band and records it in `.env`.

**Action**: repoint every consumer from `http://localhost:8100` to your
instance's `RAG_PORT` (default `http://localhost:10000`). Because `.env` values
**win** over the new defaults (see "generate_env preserves .env" below), an old
`.env` carrying `RAG_PORT=8100` will keep you on 8100 until you edit it — and
deleting `.env` does NOT cure this by itself: it is regenerated **from
`config/config.yaml`**, so an old config.yaml carrying `network.port: 8100`
pins 8100 right back (see step 4).

## Breaking change 2 — embeddings are Ollama-only by default: 384 → 768 dims

Old copies defaulted (in `app/config.py`, the compose fallbacks, and
`deploy.sh generate_env`) to `all-MiniLM-L6-v2` / **384** dims /
`sentence-transformers`. The canonical triple is now, in every fallback layer:

```
EMBEDDING_MODEL=nomic-embed-text
EMBEDDING_DIMENSION=768
EMBEDDING_BACKEND=ollama
```

- **Hard prerequisite** before first start: an external Ollama server with the
  model pulled — `ollama pull nomic-embed-text`. The stack does NOT run its own
  Ollama container. `./deploy.sh doctor` verifies reachability + model.
- **sentence-transformers is removed as the default** and is NOT installed in
  the image. It survives only as an optional backend; selecting it without the
  dependency raises an actionable error.
- **The dimension change (384 → 768) means a RE-INGEST is REQUIRED — but only
  for indexes stored at 384 dims.** Copies that were already ingesting with
  `nomic-embed-text`/768 (recent pre-upgrade `config.yaml`/`.env.example`
  already carried the new triple; only the code fallback layers were 384)
  keep a valid index — `doctor` reports the stored dimension (step 5). The
  pgvector column width is fixed at first ingest. Two guards refuse to run
  against a mismatched stored dimension instead of corrupting data:
  - deploy-side: `deploy.sh` `dim_guard` (fatal on `start`, reported by `doctor`)
  - app-side: the API startup check reads the `rag_meta` row and refuses to
    start when `meta.dimension != EMBEDDING_DIMENSION`.

  The fix is a wipe + re-index: `./deploy.sh reset` (handles the root-owned
  postgres files via a throwaway container), then `start` + `ingest` — see the
  procedure below.

## Breaking-ish change 3 — `POST /api/ingest/directory` is now async (202 + job)

Old copies returned `201` with the full `list[DocumentSchema]` synchronously —
large corpora hit the nginx 300s read-timeout cliff. The endpoint now returns
**`202 Accepted` immediately** with a pollable job:

```bash
curl -X POST http://localhost:10000/api/ingest/directory \
  -H 'Content-Type: application/json' -d '{"path": "/absolute/path/to/docs"}'
# → 202 {"job_id": "…", "status": "queued", "status_url": "/api/ingest/status/…", …}

curl http://localhost:10000/api/ingest/status/<job_id>
# poll until status is "completed" (documents list attached) or "failed" (error attached)
```

**Action: update any caller that awaited the old synchronous 201 response** to
poll `GET /api/ingest/status/{job_id}` until a terminal state. The in-repo
consumers (`deploy.sh ingest`, `scripts/bootstrap.sh`, the frontend) already
poll. Notes:

- The request body accepts optional `extensions` / `exclude` lists overriding
  `config/config.yaml`'s `ingestion.*` for that one run.
- Above 500 in-flight jobs the API refuses new ingests with `429` +
  `Retry-After` instead of growing without bound.
- Jobs are in-process: a restart forgets them (re-ingest is idempotent, so
  re-POSTing is safe).

## New endpoints (additive)

| Endpoint | What it gives you |
|----------|-------------------|
| `GET /api/status` | Corpus whoami: `project_name`, `docs_root`, `documents`, `chunks`, `embedding_model`, `dimension`, `indexed_at` — verify you're talking to the right corpus before trusting results |
| `GET /health/index` | Index freshness: `indexed_at`, `indexed_commit_sha`, `corpus_bytes`, `chunks`, `documents`, `source_root`, `staleness_class` (`unindexed`/`fresh`/`aging`/`stale`) |
| `POST /api/v1/search` (+ `/api/v1/ask`, `/api/v1/status`, `/api/v1/health`) | **Additive aliases** of the unversioned routes (same handlers/schemas). `/api/search` remains stable; new consumers should pin `/api/v1/*`. **Search is POST-only** — `GET /api/v1/search?q=…` returns `405 Method Not Allowed` |
| `GET /api/health?deep=1` | Optional deep health (embed + 1-NN round-trip) on top of the liveness check |

Search responses also carry an `X-Chunk-Bytes` telemetry header and an optional
`corpus` request field (literal filepath-prefix filter within the corpus).

## New knobs (defaults changed vs old copies)

| Knob | Default | Behavior change vs pre-2026-07-03 |
|------|---------|-----------------------------------|
| `DOCS_PATH` | `${PWD}/data/docs` | Same-path **read-only** mount, **default-on**: the container sees your docs root at the SAME absolute path as the host, so `document_filepath` citations stored by **host-path ingests** (`deploy.sh ingest`, `POST /api/ingest/directory`) open directly on the host. Caveat: the **default-on auto-ingest watcher** ingests through the `/data/docs` container mount first, so files it picks up get container-path citations (`/data/docs/…`) — see the live-smoke report, finding 3. If host-openable citations matter, ingest via host-path POST before the watcher races you, or set `WATCHER_ENABLED=false`. Point `DOCS_PATH` at your real docs root. |
| `RAG_INGEST_ROOT_GUARD` | `on` | With `DOCS_PATH` now defaulted, the guard is **active out of the box**: `/api/ingest/directory` rejects paths outside the docs root with `400`. Old copies would ingest any container-readable path. Opt out with `RAG_INGEST_ROOT_GUARD=off`. |
| `WATCHER_ARCHIVE_MODE` | `off` | **The auto-ingest watcher no longer moves your files by default.** Old copies relocated processed files into `archive/`. Modes: `off` (never touch sources), `copy`, `move` (legacy — staging dirs only). |

"Root Discipline").

## deploy.sh: what's new, and how it treats your `.env`

- **`./deploy.sh doctor`** — preflight everything: config resolution, effective
  config echo, Docker daemon, `RAG_PORT` free, Ollama reachable, embedding
  model pulled, disk space, stored-vs-configured dimension. Run it first,
  always.
- **`./deploy.sh reset`** — full wipe for re-init; removes the root-owned
  `data/docker/` postgres files via a throwaway root container (no sudo).
- **`generate_env` PRESERVES a customized `.env`**: with no `.env` it generates
  one from `config/config.yaml`; with an existing `.env` it only **appends
  missing keys** and never overwrites your values (`COMPOSE_PROJECT_NAME`,
  `RAG_NETWORK_NAME`, ports, embedding triple, …). Drift between `config.yaml`
  and `.env` is warned about, and **`.env` wins**. Consequence for upgrades:
  stale old values (`RAG_PORT=8100`, `EMBEDDING_DIMENSION=384`) survive until
  YOU edit or delete them — and deleting only helps if `config/config.yaml`
  itself carries the new values, because the regenerated `.env` copies them
  from there (an old config.yaml's `network.port: 8100` becomes
  `RAG_PORT=8100` again).

## Step-by-step upgrade procedure

Assumes your project embeds a pre-2026-07-03 copy at `yourproject/rag-bootstrap/`.

1. **Stop the old stack** (from the old copy): `./deploy.sh stop`
   (or `docker compose down`). If the old index matters to you, back up
   `data/docker/` now — step 6, when the wipe applies, destroys it.
2. **Sync the new tree over your copy**, preserving your local state:
   keep your `.env`, your `config/config.yaml` customizations, and your corpus
   (`data/docs/` or wherever your docs live). Everything under `app/`,
   `frontend/`, `scripts/`, `tests/`, `client/`, `deploy.sh`, and the compose
   files should come from the new tree. Delete the old root strays listed in
   "Moved files" above.
3. **Pull the embedding model** (hard prerequisite): `ollama pull nomic-embed-text`
4. **Edit `.env` AND `config/config.yaml`** (deploy.sh will NOT overwrite
   stale values for you):
   - `RAG_PORT=8100` → `10000` (or your instance's band; add `RAG_PORT_BASE=10000`)
   - embedding triple → `nomic-embed-text` / `768` / `ollama` (all three together)
   - set `DOCS_PATH` to your real docs root; keep `COMPOSE_PROJECT_NAME` /
     `RAG_NETWORK_NAME` instance-unique.
   - in `config/config.yaml`, align `ingestion.directories` with `DOCS_PATH`:
     every entry must be a **host** path (absolute, or relative to the repo
     root) that is equal to or under `DOCS_PATH`. No-arg `./deploy.sh ingest`
     (step 8) POSTs exactly these entries: the old default `./docs` resolves
     to `<repo>/docs`, which is **not mounted** in the api container (`400
     Directory not found`), and legacy **container-path** entries like
     `/data/docs` are rejected `400` by the now default-on ingest-root guard.
     The new canonical default is `./data/docs` (matches the default
     `DOCS_PATH`).
   Alternatively: delete `.env` entirely and let `./deploy.sh` regenerate it
   from your `config/config.yaml` — but ONLY after updating `config.yaml`
   itself, since regeneration copies its values: an old copy's
   `network.port: 8100` silently pins `RAG_PORT=8100` again and step 9's
   verify commands against `:10000` will fail. Update `config.yaml`
   (`network.port: 10000`, embedding `nomic-embed-text`/768/`ollama`,
   `ingestion.directories` as above) first, then check the regenerated `.env`
   (or the `doctor` effective-config echo) before continuing.
5. **Preflight**: `./deploy.sh doctor` — fix anything red before continuing.
6. **Wipe the index — only if step 5's doctor reported a stored-dimension
   mismatch** (`database stores 384, config wants 768`): `./deploy.sh reset`.
   If doctor said the stored dimension already matches 768 (copies that were
   already ingesting with `nomic-embed-text`/768 — see breaking change 2), or
   reported no stored embeddings yet, **skip this step**: `reset` is
   destructive and would wipe a valid index for nothing.
7. **Start**: `./deploy.sh start --build`
8. **Re-ingest** (needed after a step-6 wipe or when doctor showed no stored
   embeddings; skip only if you kept a valid 768 index): `./deploy.sh ingest`
   — no-arg ingest POSTs the `ingestion.directories` entries you aligned in
   step 4.
   Or pass a path explicitly (`./deploy.sh ingest /abs/path/under/docs-root`),
   or POST `/api/ingest/directory` and poll the job — the path must be equal
   to or under `DOCS_PATH` either way.
9. **Verify**:
   ```bash
   curl -s http://localhost:10000/api/status        # right corpus? dimension 768?
   curl -s http://localhost:10000/health/index      # staleness_class fresh?
   curl -s -X POST http://localhost:10000/api/v1/search \
     -H 'Content-Type: application/json' -d '{"query":"<known term>","limit":3}'
   ```
10. **Update your callers**: base URL port, async-ingest polling (breaking
    change 3), and pin `/api/v1/*` going forward. If your agents consume the
    stack, re-read [docs/integration/CONSUMING_AGENTS_CONTRACT.md](../integration/CONSUMING_AGENTS_CONTRACT.md)
    and `agent_hints/HOW_TO_QUERY.md`.

## Rollback

Restore your pre-upgrade copy of the directory (or `git checkout` your
project's previous vendored tree) and your `data/docker/` backup from step 1,
then `./deploy.sh start` from the old copy. The old stack serves the old
384-dim index on port 8100 exactly as before — the upgrade touches nothing
outside the `rag-bootstrap/` directory. Without the `data/docker/` backup the
old index is gone (`reset` is destructive); the corpus files themselves are
never touched, so a fresh old-version ingest also works.
