Concern: Port configuration should BASE at 10000 so downstream apps can increment
within the 10000 range to avoid local collisions.

`ports:` mapping leaves them untouched — as long as only the left/host number changes.

## 2. Is configuration centralized or scattered?
PARTIALLY centralized, mostly scattered:
- The primary web port IS centralized behind one var `RAG_PORT` (default 8100),
  threaded through .env:24, .env.example:24, config/config.yaml:36, app/config.py:49,
  both compose files, deploy.sh:45/159. Good single-knob for the one exposed port.
- BUT config/config.yaml:36 (`port: 8100`) is a SECOND source of truth for the same
  web port, separate from `.env` RAG_PORT — deploy.sh:69 reads it into CONFIG_PORT then
  writes RAG_PORT. Two files must be kept in sync.
- The ENTIRE monitoring stack (7 host ports: 9090/3000/9093/9187/9121/8080/3100) is
  HARDCODED in docker-compose.monitoring.yml with no env indirection and no relation to
  any base — pure collision risk when a second stack runs.
- scripts/health-check.sh:70-75 hardcodes 8100, 9090, 3000, 9093.
- scripts/bootstrap.sh:49-51 reads a DIFFERENT variable name `RAG_API_PORT` (not
  RAG_PORT) — inconsistent knob that will silently fall back to 8100.
- README.md hardcodes 8100 in ~10 spots (17,20,58,73,94,122,157,175, etc.).
- There is NO base-port variable anywhere. `grep RAG_PORT_BASE|PORT_BASE` = 0 hits.

## 3. Proposed BASE-at-10000 scheme (single source of truth)
Introduce ONE base var `RAG_PORT_BASE=10000` in `.env` / `.env.example`; derive every
HOST-published port as `base + fixed offset`. Downstream projects shift the whole block
by changing only `RAG_PORT_BASE` (e.g. 10000, 10100, 10200 per instance).

| Service | Offset | Proposed host port | Compose var expression |
|---|---|---|---|
| web / API (frontend nginx) | +0 | 10000 | `${RAG_PORT:-10000}` |
| prometheus | +10 | 10010 | `${RAG_PROMETHEUS_PORT:-10010}` |
| grafana | +11 | 10011 | `${RAG_GRAFANA_PORT:-10011}` |
| loki | +12 | 10012 | `${RAG_LOKI_PORT:-10012}` |
| alertmanager | +13 | 10013 | `${RAG_ALERTMANAGER_PORT:-10013}` |
| postgres-exporter | +14 | 10014 | `${RAG_PG_EXPORTER_PORT:-10014}` |
| redis-exporter | +15 | 10015 | `${RAG_REDIS_EXPORTER_PORT:-10015}` |
| cadvisor | +16 | 10016 | `${RAG_CADVISOR_PORT:-10016}` |

Only the HOST (left) side of each `ports:` mapping changes; container (right) side stays
(9090:9090 -> `10010:9090`), so prometheus.yml scrape targets and all in-network DNS
remain valid with zero edits.

Reserve 10000-10019 for this stack (20 slots); +20.. free for downstream app growth.

Notes / caveats for the implementer:
- Compose cannot do arithmetic on env vars. Two viable single-knob patterns:
  (a) keep RAG_PORT_BASE as documentation and define each RAG_<svc>_PORT explicitly in
      .env (pre-computed offsets), OR
  (b) have deploy.sh compute `RAG_<svc>_PORT = BASE + offset` and write them into .env
      (mirrors how deploy.sh already generates RAG_PORT from config.yaml at line 159).
  Recommend (b) so a single `RAG_PORT_BASE` edit truly shifts everything.
- alertmanager `--web.external.url=http://localhost:9093` (compose.monitoring.yml:87)
  must track the new host port or generated links break.
- Internal-only ports (5432, 6379, 8000, 80, promtail 9080, loki listen 3100) need NOT
  move — they never touch the host. Leaving them fixed keeps app/config.py, nginx.conf,
  Dockerfiles, prometheus.yml untouched. Only remap them if a future need publishes them.
- Ollama 11434 must stay (external well-known host service).
- Fix the two consistency defects while here: config/config.yaml:36 duplicate port
  source, and scripts/bootstrap.sh RAG_API_PORT vs RAG_PORT name mismatch.
