## SLO Definitions

Service Level Objectives define acceptable performance thresholds for v12.9.0 production deployment.

### Availability SLO

**Objective:** 99.9% uptime (43.2 minutes max downtime per month)

| Metric | Target | Warning | Critical |
|--------|--------|---------|----------|
| API Availability | ≥99.9% | <99.5% | <99.0% |
| Health Check Pass Rate | ≥99.5% | <98.5% | <98.0% |
| Container Restart Count | 0/week | >2/week | >5/week |
| Process Crash Rate | 0/day | >1/week | >3/week |

**Calculation:**
```
Availability = (Successful Requests / Total Requests) × 100
Target = Requests responding with 200-299 / Total Requests
```

**Monitoring:**
- Health check endpoint: `/api/diagnostics` every 30 seconds
- Docker container health status
- Process uptime tracking
- Connection establishment success rate

### Latency SLO

**Objective:** 95% of requests complete within 100ms

| Percentile | Target | Warning | Critical |
|------------|--------|---------|----------|
| P50 (Median) | ≤50ms | >75ms | >150ms |
| P95 | ≤100ms | >150ms | >300ms |
| P99 | ≤200ms | >350ms | >600ms |
| P99.9 | ≤500ms | >800ms | >1500ms |

**Latency Breakdown by Command:**

| Command Type | P50 | P95 | P99 |
|--------------|-----|-----|-----|
| Navigate | 80ms | 150ms | 250ms |
| Click | 30ms | 60ms | 100ms |
| Fill | 25ms | 50ms | 80ms |
| Screenshot | 100ms | 200ms | 400ms |
| Extract HTML | 40ms | 100ms | 200ms |
| Execute JS | 50ms | 120ms | 250ms |

**Monitoring:**
- Histogram metrics for duration distribution
- Percentile calculations (P50, P95, P99, P99.9)
- Command-specific latency tracking
- Database query performance (if applicable)

### Throughput SLO

**Objective:** Minimum 100 msg/sec sustainable throughput

| Metric | Target | Warning | Critical |
|--------|--------|---------|----------|
| Baseline Throughput | ≥100 msg/sec | <150 msg/sec | <80 msg/sec |
| Peak Throughput | ≥500 msg/sec | <400 msg/sec | <250 msg/sec |
| Request Queue Depth | <100 | >200 | >500 |
| Message Drop Rate | 0% | >0.1% | >0.5% |

**Monitoring:**
- Messages/second counter
- Request queue depth gauge
- Message drop tracking
- Backlog accumulation rate

### Additional SLOs

#### Error Rate
- **Objective:** <0.1% error rate
- **Warning:** >0.05% errors in 5m window
- **Critical:** >0.1% errors in 1m window
- **Excludes:** Client-side errors (4xx), intentional failures

#### Resource Utilization
- **Memory:** <75% heap usage (warning), <90% (critical)
- **CPU:** <70% sustained (warning), <85% (critical)
- **Disk:** <80% usage (warning), <95% (critical)
- **Connections:** <80% of max (warning), <95% (critical)

## Prometheus Metrics

### WebSocket Connection Metrics

```prometheus
# Active WebSocket connections
basset_websocket_connections_active

# Total connections created (counter)
basset_websocket_connections_total_created

# Total connections closed (counter)
basset_websocket_connections_total_closed

# Connection errors (counter)
basset_websocket_connection_errors_total

# Connection duration percentiles (milliseconds)
basset_websocket_connection_duration_ms{quantile="0.5"}  # P50
basset_websocket_connection_duration_ms{quantile="0.95"}  # P95
basset_websocket_connection_duration_ms{quantile="0.99"}  # P99
```

### Command Execution Metrics

```prometheus
# Total command executions by command name
basset_command_executions_total{command="navigate"}
basset_command_executions_total{command="click"}
basset_command_executions_total{command="fill"}
# ... (164 total commands)

# Command duration metrics (milliseconds)
basset_command_duration_ms{command="navigate", quantile="avg"}
basset_command_duration_ms{command="navigate", quantile="0.95"}
basset_command_duration_ms{command="navigate", quantile="min"}
basset_command_duration_ms{command="navigate", quantile="max"}

# Command error metrics
basset_command_errors_total{command="navigate"}
basset_command_error_rate{command="navigate"}  # Percentage
```

# Messages sent/received

## Grafana Dashboards

Three primary dashboards provide comprehensive v12.9.0 monitoring.

### Dashboard 1: Operations Overview

**Title:** Basset Hound Browser - Operations  
**Refresh:** 30 seconds  
**Time Range:** Last 6 hours  
**Target Audience:** Operations, DevOps  

**Panels:**

1. **Status Summary** (Top row)
   - Service Health: Green (1) / Red (0)
   - Uptime: Duration display
   - Active Connections: Big number display
   - Last Check: Time display
   - Query: `basset_health_check_status`, `basset_process_uptime_seconds`, `basset_websocket_connections_active`

2. **Throughput (msg/sec)** 
   - Type: Graph
   - Size: 12w × 8h (top-left)
   - Query: `rate(basset_websocket_messages_sent_total[1m])`
   - Axes: Y = Messages/sec
   - Alert threshold: 100 msg/sec (warning line)

3. **Latency Percentiles**
   - Type: Graph
   - Size: 12w × 8h (top-right)
   - Queries:
     - `histogram_quantile(0.50, basset_command_duration_ms)` → P50
     - `histogram_quantile(0.95, basset_command_duration_ms)` → P95
     - `histogram_quantile(0.99, basset_command_duration_ms)` → P99
   - Axes: Y = Milliseconds
   - Thresholds: 50ms (green), 100ms (yellow), 200ms (red)

4. **Error Rate**
   - Type: Graph
   - Size: 12w × 8h
   - Query: `rate(basset_command_errors_total[5m]) / rate(basset_command_executions_total[5m])`
   - Axes: Y = Percentage
   - Thresholds: 0.05% (yellow), 0.1% (red)

5. **Memory Usage**
   - Type: Graph
   - Size: 12w × 8h
   - Queries:
     - `basset_process_heap_used_bytes / basset_process_heap_total_bytes * 100` → Heap %
     - `(basset_system_memory_total_bytes - basset_system_memory_free_bytes) / basset_system_memory_total_bytes * 100` → System %
   - Axes: Y = Percentage (0-100)
   - Thresholds: 75% (yellow), 90% (red)

6. **Active Connections**
   - Type: Stat (big number)
   - Size: 6w × 4h
   - Query: `basset_websocket_connections_active`
   - Thresholds: 100 (green), 200 (yellow), 300+ (red)

7. **Connection Duration Distribution**
   - Type: Heatmap
   - Size: 24w × 8h
   - Query: `sum(rate(basset_websocket_connection_duration_ms[5m])) by (le)`
   - Color: Blue (cool) → Red (hot)

**Title:** Basset Hound Browser - Command Performance  
**Refresh:** 30 seconds

**Target Audience:** Dev team, Performance engineers

   - Type: Graph
   - Size: 12w × 6h

   - Type: Graph
   - Size: 12w × 6h

5. **Command Error Rate Trend**

   - Size: 24w × 8h

### Dashboard 3: Infrastructure & System Health

**Title:** Basset Hound Browser - System Health  
**Refresh:** 30 seconds  
**Time Range:** Last 24 hours  
**Target Audience:** SRE, Infrastructure team  

**Panels:**

1. **System Load Average**
   - Type: Graph
   - Size: 12w × 8h
   - Queries:
     - `basset_system_load_average{interval="1m"}` → 1m avg
     - `basset_system_load_average{interval="5m"}` → 5m avg
     - `basset_system_load_average{interval="15m"}` → 15m avg
   - Reference line: CPU core count

2. **CPU Cores Available**
   - Type: Stat
   - Size: 6w × 4h
   - Query: `basset_system_cpu_cores`

3. **Memory Distribution**
   - Type: Pie chart
   - Size: 6w × 8h
   - Queries:
     - Free: `basset_system_memory_free_bytes`
     - Used: `basset_system_memory_used_bytes`
   - Shows system memory allocation

4. **Heap Memory Timeline**
   - Type: Graph
   - Size: 12w × 8h
   - Queries:
     - Used: `basset_process_heap_used_bytes / 1024 / 1024` → MB
     - Total: `basset_process_heap_total_bytes / 1024 / 1024` → MB
   - Axes: Y = Megabytes
   - Thresholds: Alert if used > 1.5GB

5. **Process Uptime**
   - Type: Stat (uptime format)
   - Size: 6w × 4h
   - Query: `basset_process_uptime_seconds`

6. **Health Check Status Timeline**
   - Type: Graph
   - Size: 24w × 6h
   - Query: `basset_health_check_status`
   - Thresholds: 1.0 (green), 0.5 (yellow), 0.0 (red)

7. **Heartbeat Tracking**
   - Type: Stat
   - Size: 6w × 4h
   - Query: `basset_health_last_heartbeat_ms_ago`
   - Threshold: Alert if >60000ms (1 minute)

          slo: availability
        annotations:
          summary: "Basset Hound Browser service is down"
          description: "Service has been down for {{ $value }} minutes"

### Alert Routing Configuration

**Notification destinations:**

```yaml
# alertmanager.yml snippet
route:
  receiver: 'default'
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 12h

  routes:
    # CRITICAL alerts → PagerDuty + Slack
    - match:
        severity: critical
      receiver: 'critical'
      continue: true
      repeat_interval: 5m
      group_wait: 0s

    # WARNING alerts → Slack only
    - match:
        severity: warning
      receiver: 'warning'
      repeat_interval: 1h

receivers:
  - name: 'default'
    slack_configs:
      - api_url: '${SLACK_WEBHOOK_URL}'
        channel: '#basset-hound-alerts'
        send_resolved: true

  - name: 'critical'
    pagerduty_configs:
      - service_key: '${PAGERDUTY_SERVICE_KEY}'
    slack_configs:
      - api_url: '${SLACK_WEBHOOK_URL}'
        channel: '#basset-hound-critical'
        send_resolved: true

  - name: 'warning'
    slack_configs:
      - api_url: '${SLACK_WEBHOOK_URL}'
        channel: '#basset-hound-warnings'
        send_resolved: true
```

### Log Transports

**Primary transports:**

1. **Console (stdout)**
   - Format: JSON (structured)
   - Level: Configured by `LOG_LEVEL`
   - Always enabled in container

2. **File (rolling)**
   - Path: `/app/logs/basset-hound.log`
   - Format: JSON
   - Rotation: 10MB per file, 10 files max
   - Retention: 30 days

3. **Syslog (optional)**
   - Facility: local0
   - Protocol: UDP
   - Format: Structured (JSON)

## Runbook

### Alert Response Procedures

#### CRITICAL: ServiceUnavailable

**Severity:** CRITICAL  
**Alert:** `up{job="basset-hound-browser"} == 0`  

**Response Steps:**

1. **Immediate (30 seconds)**
   ```bash
   # Check service status
   curl -f http://localhost:8765/api/diagnostics || echo "Service Down"
   
   # Check container logs
   docker logs basset-hound-browser | tail -50
   
   # Verify network connectivity
   docker network inspect basset-hound-browser
   ```

2. **Investigation (5 minutes)**
   - Check process status: `docker ps | grep basset-hound-browser`
   - Review last 30 seconds of logs for errors
   - Check system resources: CPU, memory, disk
   - Verify port 8765 is accessible

3. **Resolution**
   ```bash
   # Option 1: Restart container
   docker restart basset-hound-browser
   
   # Option 2: Full redeploy if restart fails
   docker-compose down
   docker-compose up -d
   
   # Option 3: Manual intervention
   # See DEPLOYMENT-PROCEDURES.md for manual rollback
   ```

4. **Verification**
   ```bash
   # Wait 30 seconds for startup
   sleep 30
   
   # Verify service is responding
   curl -f http://localhost:8765/api/diagnostics
   
   # Monitor for 5 minutes
   watch -n 5 'curl http://localhost:8765/metrics | grep basset_health'
   ```

#### CRITICAL: P99LatencyExceeded

**Severity:** CRITICAL  
**Alert:** `histogram_quantile(0.99, ...) > 600ms`  

**Response Steps:**

1. **Immediate (1 minute)**
   ```bash
   # Check Grafana dashboard
   # Navigate to: http://grafana:3000/d/basset-command-perf
   
   # Identify slow commands
   curl http://localhost:8765/metrics | grep "basset_command_duration_ms{command"
   ```

2. **Investigation (5 minutes)**
   - Identify which command(s) are slow
   - Check system resources (CPU, memory, disk)
   - Review Grafana "Command Performance" dashboard
   - Look for correlated events (deployments, load increases)

3. **Mitigation**
   ```bash
   # Temporarily reduce load
   # Adjust rate limits in websocket/server.js
   # Restart with conservative limits
   
   # Or scale resources
   docker update --memory 4G basset-hound-browser
   docker restart basset-hound-browser
   ```

4. **Root Cause Analysis**
   - Check for memory leaks (monitor heapUsed over time)
   - Analyze slow command patterns
   - Profile specific slow operations
   - Review recent code changes (if applicable)

#### WARNING: HighMemoryUsage

**Severity:** WARNING  
**Alert:** `heap_used / heap_total > 75%`  

**Response Steps:**

1. **Monitor (10 minutes)**
   - Is memory usage growing? (Check Grafana)
   - Or is it stable at high level?
   - Growth indicates potential leak

2. **If Stable:**
   - No action needed
   - Memory is baseline for current load

3. **If Growing:**
   ```bash
   # Force garbage collection
   # (via metrics endpoint or manual trigger)
   
   # Monitor growth rate
   watch -n 5 'curl http://localhost:8765/metrics | grep heap'
   
   # If growth continues after GC:
   # Escalate to CRITICAL memory leak investigation
   ```

4. **Long-term Resolution**
   - Increase heap size: `NODE_OPTIONS=--max-old-space-size=4096`
   - Profile application for memory leaks
   - Review object retention patterns

#### CRITICAL: MessageDropDetected

**Severity:** CRITICAL  
**Alert:** `increase(websocket_message_errors_total[5m]) > 5`  

**Response Steps:**

1. **Immediate (30 seconds)**
   ```bash
   # Check connection status
   curl http://localhost:8765/metrics | grep websocket
   
   # Review error logs
   docker logs basset-hound-browser | grep -i error
   ```

2. **Investigation**
   - Are clients reconnecting properly?
   - Check network stability
   - Review message sizes (may exceed limits)
   - Check rate limiter status

3. **Resolution**
   ```bash
   # Restart WebSocket server
   docker restart basset-hound-browser
   
   # Or graceful restart with drain
   # Send shutdown signal and allow existing connections to drain
   kill -SIGTERM $(docker inspect -f '{{.State.Pid}}' basset-hound-browser)
   ```

#### WARNING: RequestQueueBacklogAccumulating

**Severity:** WARNING  
**Alert:** `increase(queue_depth[5m]) > 100`  

**Response Steps:**

1. **Assess Load**
   ```bash
   # Check active connections
   curl http://localhost:8765/metrics | grep connections_active
   
   # Check throughput
   curl http://localhost:8765/metrics | grep messages_sent_total
   ```

2. **Scaling Decision**
   - Can load decrease naturally? (seasonal traffic)
   - Need more capacity?
   - Connection limit too low?

3. **Scale Resources**
   ```bash
   # Increase memory
   docker update --memory 4G basset-hound-browser
   
   # Increase CPU
   docker update --cpus 4.0 basset-hound-browser
   
   # Restart
   docker restart basset-hound-browser
   ```
