## Executive Summary

This document provides a complete deployment plan for v12.1.0, covering the transition from v12.0.0. The release includes Optimization Sprint 2 (OPT-03, OPT-04, OPT-10) with significant performance improvements and zero breaking changes.

**Key Points:**
- Direct upgrade from v12.0.0 (no intermediate versions)
- Zero data migration (100% backward compatible)
- Rolling deployment possible (no coordinated shutdown)
- Performance improvements visible immediately
- Complete rollback possible (return to v12.0.0 if needed)

## Canary Deployment Procedure

### Phase 1: Preparation (30 minutes before)

1. **Verify current state:**
   ```bash
   # Check v12.0.0 is running and healthy
   docker ps | grep basset-hound-v12
   # Expected: basset-hound-v12.0.0 running
   
   # Check port 8765 accessible
   curl -i http://localhost:8765/health
   # Expected: 200 OK or ping response
   ```

2. **Pull v12.1.0 image:**
   ```bash
   docker pull basset-hound-browser:v12.1.0
   # Verify image size: ~950MB
   docker images | grep basset-hound-browser:v12.1.0
   ```

3. **Backup current configuration:**
   ```bash
   docker inspect basset-hound-v12.0.0 > /backup/v12.0.0-config.json
   docker logs basset-hound-v12.0.0 > /backup/v12.0.0-logs.tar.gz
   ```

### Phase 2: Canary Deployment (Parallel instance)

1. **Start v12.1.0 on different port (for testing):**
   ```bash
   docker run -d -p 8766:8765 \
     --name basset-hound-v12.1.0-canary \
     --memory 2g \
     --cpus 2 \
     -e BASSET_SCREENSHOT_POOL_SIZE=3 \
     -e BASSET_RECORDING_MEMORY_LIMIT=10 \
     -e BASSET_PRIORITY_QUEUE_ENABLED=true \
     basset-hound-browser:v12.1.0
   ```

2. **Wait for startup:**
   ```bash
   # Wait for service to be ready (30-60 seconds)
   sleep 30
   docker logs basset-hound-v12.1.0-canary | tail -20
   ```

3. **Run smoke tests:**
   ```bash
   # Ping test
   curl -X POST http://localhost:8766 \
     -H "Content-Type: application/json" \
     -d '{"type":"ping"}'
   # Expected: {"status":"ok"}
   
   # Screenshot test
   curl -X POST http://localhost:8766 \
     -H "Content-Type: application/json" \
     -d '{"type":"navigate","url":"https://example.com"}' && \
   curl -X POST http://localhost:8766 \
     -H "Content-Type: application/json" \
     -d '{"type":"screenshot"}'
   # Expected: screenshot data
   ```

### Phase 3: Canary Validation (15-30 minutes)

1. **Performance baseline:**
   ```bash
   # Log current metrics from v12.0.0
   docker stats basset-hound-v12.0.0 --no-stream
   
   # Log metrics from v12.1.0 canary
   docker stats basset-hound-v12.1.0-canary --no-stream
   ```

2. **Memory comparison:**
   ```bash
   # Check memory usage (should be similar at start, but growth slower over time)
   # Monitor for 5 minutes
   watch -n 5 'docker stats basset-hound-v12.1.0-canary --no-stream | grep basset'
   ```

3. **Latency testing:**
   ```bash
   # Create test script to verify P95 latency < 500ms for mixed workload
   # (See validation scripts below)
   ```

4. **Log review:**
   ```bash
   # Check for any errors or warnings
   docker logs basset-hound-v12.1.0-canary | grep -i "error\|warning\|critical"
   # Should be minimal or none
   ```

### Phase 4: Canary Approval

If all checks pass:
- [ ] Smoke tests passed

## Progressive Rollout Procedure

### Wave 1: 20% Deployment (June 15, 10:00 AM)

**Target:** 2 of 10 instances

**Procedure:**
1. Identify 2 instances for Wave 1 (prefer non-critical first)
2. For each instance:
   ```bash
   # 1. Stop old version gracefully (wait for requests to complete)
   docker stop -t 30 basset-hound-v12.0.0
   
   # 2. Backup logs
   docker logs basset-hound-v12.0.0 >> /backup/v12.0.0-instance-logs.txt 2>&1
   
   # 3. Remove old container
   docker rm basset-hound-v12.0.0
   
   # 4. Start new version with same port
   docker run -d -p 8765:8765 \
     --name basset-hound-v12.1.0 \
     --memory 2g \
     --cpus 2 \
     -e BASSET_SCREENSHOT_POOL_SIZE=3 \
     -e BASSET_RECORDING_MEMORY_LIMIT=10 \
     -e BASSET_PRIORITY_QUEUE_ENABLED=true \
     basset-hound-browser:v12.1.0
   
   # 5. Wait for startup
   sleep 30
   
   # 6. Verify health
   curl -i http://localhost:8765/health
   ```

3. **Monitor for 1 hour:**
   - Check memory usage every 10 minutes
   - Monitor error logs
   - Verify latency metrics
   - Confirm no connection drops

4. **Decision:**
   - If healthy: Proceed to Wave 2
   - If issues: Rollback (see rollback procedure)

### Wave 2: 40% Deployment (June 15, 2:00 PM)

**Target:** 4 of 10 instances (40% total, includes Wave 1)

**Procedure:**
- Repeat Wave 1 procedure for 2 additional instances
- Monitor for 1 hour after each instance
- Cumulative uptime target: 2+ hours at 40% capacity

### Wave 3: 100% Deployment (June 16, 9:00 AM)

**Target:** Remaining 6 instances (100% total)

**Procedure:**
- Repeat Wave 1 procedure for remaining 6 instances
- Deploy in pairs (3 deployments of 2 instances each)
- Each deployment spaced 15-30 minutes apart
- Monitor after each pair

## Validation Test Scripts

### Script 1: Performance Comparison (40 lines)

```bash
#!/bin/bash
# performance-validation.sh

echo "=== Performance Validation ==="
echo "Testing concurrent screenshots..."

# Test 10 concurrent screenshots
START=$(date +%s%N)
for i in {1..10}; do
  (curl -X POST http://localhost:8765 \
    -H "Content-Type: application/json" \
    -d '{"type":"navigate","url":"https://example.com"}' > /dev/null 2>&1 &
   curl -X POST http://localhost:8765 \
    -H "Content-Type: application/json" \
    -d '{"type":"screenshot"}' > /dev/null 2>&1 &)
done
wait
END=$(date +%s%N)
ELAPSED=$((($END - $START) / 1000000))

echo "10 screenshots completed in: ${ELAPSED}ms"
if [ $ELAPSED -lt 1000 ]; then
  echo "✅ PASS: Performance improved (target: <1000ms)"
else
  echo "❌ FAIL: Performance regression (target: <1000ms)"
fi
```

### Script 2: Memory Monitoring (35 lines)

```bash
#!/bin/bash
# memory-validation.sh

echo "=== Memory Monitoring (5 minute baseline) ==="

# Get initial memory
INITIAL=$(docker stats basset-hound-v12.1.0 --no-stream --format "{{.MemUsage}}" | grep -oE "[0-9]+\.[0-9]+")
echo "Initial memory: ${INITIAL}MB"

# Monitor for 5 minutes
for i in {1..5}; do
  sleep 60
  CURRENT=$(docker stats basset-hound-v12.1.0 --no-stream --format "{{.MemUsage}}" | grep -oE "[0-9]+\.[0-9]+")
  GROWTH=$(echo "$CURRENT - $INITIAL" | bc)
  echo "After ${i} min: ${CURRENT}MB (growth: ${GROWTH}MB)"
done

# Final check
FINAL=$(docker stats basset-hound-v12.1.0 --no-stream --format "{{.MemUsage}}" | grep -oE "[0-9]+\.[0-9]+")
TOTAL_GROWTH=$(echo "$FINAL - $INITIAL" | bc)

echo ""
echo "Total growth: ${TOTAL_GROWTH}MB in 5 minutes"
if (( $(echo "$TOTAL_GROWTH < 50" | bc -l) )); then
  echo "✅ PASS: Memory growth acceptable"
else
  echo "❌ FAIL: Excessive memory growth"
fi
```

### Script 3: Latency Percentiles (45 lines)

```bash
#!/bin/bash
# latency-validation.sh

echo "=== Latency Percentile Analysis ==="

# Create test requests
declare -a latencies

for i in {1..100}; do
  START=$(date +%s%N)
  
  # Alternating request types
  if [ $((i % 3)) -eq 0 ]; then
    # Critical (screenshot)
    curl -s -X POST http://localhost:8765 \
      -H "Content-Type: application/json" \
      -d '{"type":"screenshot"}' > /dev/null
  elif [ $((i % 3)) -eq 1 ]; then
    # Normal (navigate)
    curl -s -X POST http://localhost:8765 \
      -H "Content-Type: application/json" \
      -d '{"type":"navigate","url":"https://example.com"}' > /dev/null
  else
    # Low (ping)
    curl -s -X POST http://localhost:8765 \
      -H "Content-Type: application/json" \
      -d '{"type":"ping"}' > /dev/null
  fi
  
  END=$(date +%s%N)
  LATENCY=$(( ($END - $START) / 1000000 ))
  latencies+=($LATENCY)
done

# Sort and calculate percentiles
IFS=$'\n' sorted=($(sort -n <<<"${latencies[*]}"))
P50=${sorted[50]}
P95=${sorted[95]}
P99=${sorted[99]}

echo "P50: ${P50}ms"
echo "P95: ${P95}ms"
echo "P99: ${P99}ms"

if [ $P95 -lt 500 ]; then
  echo "✅ PASS: P95 latency excellent"
elif [ $P95 -lt 1000 ]; then
  echo "✅ PASS: P95 latency acceptable"
else
  echo "❌ FAIL: P95 latency too high"
fi
```

## Rollback Procedure

### Quick Rollback (If needed during deployment)

**Timing:** Should be executed immediately upon issue detection

1. **Stop v12.1.0 instance:**
   ```bash
   docker stop -t 10 basset-hound-v12.1.0
   ```

2. **Restart v12.0.0 from backup:**
   ```bash
   docker rm basset-hound-v12.1.0
   
   docker run -d -p 8765:8765 \
     --name basset-hound-v12.0.0 \
     basset-hound-browser:v12.0.0
   ```

3. **Verify restoration:**
   ```bash
   sleep 30
   curl -i http://localhost:8765/health
   # Expected: 200 OK
   ```

4. **Validate:**
   ```bash
   # Run same validation script as Checkpoint 1
   docker stats basset-hound-v12.0.0 --no-stream
   ```

### Partial Rollback (If some instances have issues)

If deployment is at Wave 2 or Wave 3 with some instances failing:

1. **Identify affected instances** (1-2 at most)
2. **Rollback only those instances** using quick rollback procedure
3. **Investigate root cause** while keeping successful instances on v12.1.0
4. **Fix and redeploy** failed instances after root cause addressed

## Success Criteria

### Deployment Success

Deployment is considered successful if:

1. **All instances healthy:**
   - ✅ All 10 instances running v12.1.0
   - ✅ All responding to requests
   - ✅ No cascading failures

2. **No performance regression:**
   - ✅ P95 latency improved (target: <500ms, was ~2000ms)
   - ✅ Throughput maintained or improved
   - ✅ Error rate <0.1%

3. **Performance improvements realized:**
   - ✅ Screenshot throughput 2-3x faster
   - ✅ Memory growth 50-80% reduction for long sessions
   - ✅ Priority queue reducing P95 for low-priority ops

4. **Stability verified:**
   - ✅ 4+ hours stable operation after Wave 3
   - ✅ No memory leaks detected
   - ✅ Log review shows no critical issues

### Deployment Failure Criteria

Rollback immediately if:

1. **Critical system failures:**
   - Crashes on startup
   - Inability to accept requests
   - Data corruption

2. **Performance regression:**
   - P95 latency >2x worse than v12.0.0
   - Memory growth >10MB/minute sustained
   - Error rate >1%

3. **Operational issues:**
   - Can't rollback
   - Cascading failures across instances
   - Unrecoverable state

## Escalation Procedures

### Issue Severity: CRITICAL (Immediate action)

**When:** System completely unavailable

**Response:**
1. Initiate rollback immediately (via quick rollback procedure)
2. Notify all stakeholders
3. Begin root cause analysis after restoration

**Contact:** On-call engineer → Engineering Manager → VP Engineering

### Issue Severity: HIGH (Within 30 minutes)

**When:** Multiple instances failing OR >50% error rate

**Response:**
1. Escalate to on-call engineer + engineering manager
2. Decide: Continue deployment or investigate
3. If investigating: Hold deployment, analyze, fix, retest

**Contact:** On-call engineer → Engineering Manager

### Issue Severity: MEDIUM (Within 2 hours)

**When:** Single instance failing OR performance not meeting targets

**Response:**
1. Skip that instance, continue with others
2. Investigate root cause
3. Determine fix (deploy fix or rollback instance)

**Contact:** On-call engineer

### Issue Severity: LOW (During standard hours)

**When:** Warnings, minor issues, non-critical observations

**Response:**
1. Log issue
2. Investigate during post-deployment analysis
3. Plan fix for future

**Contact:** Engineering team

## Contingency Plans

### Contingency 1: Docker Image Build Fails

**If:** Docker image can't be built on deployment day

**Response:**
1. Use pre-built image from June 10 build (should exist)
2. If no pre-built image: Delay deployment by 1 day
3. Fix build issue in parallel

### Contingency 2: Instance Fails to Start

**If:** One or more instances won't start v12.1.0

**Response:**
1. Try again (occasional startup failures are normal)
2. If persistent: Check logs for startup errors
3. Restore v12.0.0 for that instance
4. Investigate startup issue post-deployment

### Contingency 3: Performance Worse Than Expected

**If:** v12.1.0 shows worse performance than v12.0.0

**Response:**
1. Verify test accuracy (compare identical workloads)
2. Adjust new environment variables (pool size, limits)
3. If still worse: Rollback and investigate
4. Performance analysis post-mortem

### Contingency 4: Data Inconsistency

**If:** v12.1.0 reports different results than v12.0.0 for same input

**Response:**
1. This should not happen (100% backward compatible)
2. If happens: Immediate rollback
3. Priority 1 investigation (potential data corruption)

### Contingency 5: Extended Deployment

**If:** Deployment takes longer than planned (issues, fixes, etc.)

**Response:**
1. Extend deployment window by 1 day if needed
2. No pressure to force completion
3. Complete Wave 3 next business day
4. Communicate delays to stakeholders
