## Executive Summary

This guide establishes procedures for maintaining forensic chain of custody when using Basset Hound Browser for OSINT investigations that may be used as evidence in legal proceedings. Proper chain of custody is critical for admissibility in court and investigative credibility.

**Key Principle:** Document everything. Authenticity and reliability depend on meticulous record-keeping.

---

## Legal Admissibility Framework

### Admissibility Standards by Jurisdiction

#### United States (Federal Rules of Evidence)
- **FRE 901(a):** Evidence must be authenticated
- **FRE 1002:** Original or duplicate acceptable if reliable
- **Daubert Standard:** Expert testimony on data collection methods
- **Relevance (FRE 401-403):** Evidence must be relevant and probative

#### EU (GDPR & eIDAS)
- **eIDAS Regulation:** Electronic evidence integrity requirements
- **GDPR:** Data collection transparency and legality
- **National Laws:** Vary by member state

#### Common Law (UK, Commonwealth)
- **Digital Evidence Guidelines:** Chain of custody requirements
- **Computer Misuse Act 1990:** Lawful access verification

### Court Acceptance Criteria

For evidence collected via Basset Hound Browser to be admissible in court:

1. **Authenticity:** Proven to be what it claims to be
2. **Reliability:** Data collection method demonstrated as reliable
3. **Integrity:** No alteration or corruption during collection
4. **Completeness:** All relevant metadata preserved
5. **Legality:** Collection method legal and authorized
6. **Documentation:** Complete chain of custody documented

## Chain of Custody Requirements

### Core Elements

Every piece of digital evidence must include:

1. **Unique Identifier**
   - Evidence number (e.g., EV-2026-05-31-001)
   - Case reference (e.g., CASE-2024-12345)
   - Date and time collected (UTC, with timezone notation)

2. **Custody Documentation**
   - Collector identity (name, badge/ID, agency)
   - Recipient identity (name, badge/ID, agency)
   - Date/time of each transfer
   - Purpose of transfer
   - Signatures/initials on each transfer

3. **Evidence Description**
   - Type (screenshot, HTML, metadata, video)
   - Source (target URL, timestamp)
   - Location (file path, storage medium)
   - File size and hash
   - Collection method used

4. **Handling Instructions**
   - Storage conditions (temperature, humidity, security)
   - Access restrictions
   - Preservation requirements
   - Disposal instructions

5. **Modification Log**
   - Any analysis performed
   - Tools used and versions
   - Date/time of analysis
   - Analyst identity and credentials

## Technical Implementation

### Hash Verification (Critical for Integrity)

All collected evidence must be cryptographically hashed:

```bash
# Generate SHA-256 hash for collected evidence
sha256sum evidence-2026-05-31-001.json > evidence-2026-05-31-001.sha256

# Example output:
# a1b2c3d4e5f6... (64 hex characters) evidence-2026-05-31-001.json

# Verify hash has not changed
sha256sum -c evidence-2026-05-31-001.sha256
# Output: evidence-2026-05-31-001.json: OK
```

**Timestamps must be:**
- Accurate to at least second precision (preferably millisecond)
- Always in UTC with timezone notation
- Consistent across all documentation
- Never adjusted retroactively

## Collection Procedures

### Step 1: Preparation Phase

Before any collection:

```
☐ Obtain search warrant (if required by jurisdiction)
☐ Establish chain of custody documentation template
☐ Verify legal authority for collection
☐ Prepare evidence storage and labeling
☐ Document investigative objective
☐ Identify authorized personnel
☐ Establish access control procedures
☐ Prepare backup and redundancy systems
```

### Step 2: Collection Phase

During evidence collection:

```
☐ Start new evidence log entry
☐ Document investigator identity and credentials
☐ Record precise timestamp (UTC)
☐ Document target URL or identifier
☐ Execute collection command
☐ Immediately hash resulting evidence file
☐ Document evidence file details (size, format, etc.)
☐ Verify hash matches expected value
☐ Store in restricted access location
☐ Complete evidence form (see template below)
```

### Step 3: Verification Phase

Immediately after collection:

```
☐ Verify file integrity (hash check)
☐ Review file metadata
☐ Check for collection errors
☐ Document any anomalies
☐ Obtain supervisor signature/initials
☐ Create backup copy
☐ Verify backup integrity
☐ Store backup in separate location
☐ Document backup location and hash
```

### Step 4: Storage Phase

Long-term evidence storage:

```
☐ Store in restricted access area
☐ Maintain climate control (if physical media)
☐ Document storage location and conditions
☐ Establish access log for any review
☐ Create audit trail for access
☐ Plan for long-term preservation
☐ Document any changes or handling
```

## Evidence Collection Workflow

### Collection Command with Evidence Documentation

**Recommended WebSocket command structure:**

```javascript
// 1. Initiate collection with full metadata
const collectionRequest = {
  id: "evidence-001",
  command: "execute_screenshot",
  
  // Target information
  url: "https://example.com/investigation",
  
  // Evidence metadata
  evidence_metadata: {
    case_reference: "CASE-2024-12345",
    investigator: "Detective John Smith",
    investigator_id: "12345",
    agency: "FBI Cyber Division",
    authorization: "Search Warrant #2024-SW-98765",
    legal_basis: "18 U.S.C. § 2703(b)"
  },
  
  // Preservation requirements
  capture_metadata: true,
  preserve_headers: true,
  record_timing: true,
  hash_algorithm: "SHA-256"
};

// 2. Send collection command
const response = await sendCommand(collectionRequest);

// 3. Process response with chain of custody
if (response.success) {
  const evidence = response.data;
  
  // Immediate verification
  const hash = computeHash(evidence.file_content);
  
  // Documentation
  const chainOfCustody = {
    evidence_id: `EV-${getTimestampISO()}-${generateNonce()}`,
    case_reference: collectionRequest.evidence_metadata.case_reference,
    collection_timestamp: response.timestamp,
    collector: collectionRequest.evidence_metadata.investigator,
    target_url: collectionRequest.url,
    file_hash: hash,
    file_size: evidence.file_content.length,
    verified_at: new Date().toISOString(),
    verified_by: process.env.ANALYST_NAME
  };
  
  // Store both evidence and documentation
  await storeEvidence(evidence.file_content, chainOfCustody);
  
  // Log for audit trail
  await logEvidenceCollection(chainOfCustody);
}
```

## Digital Signature & Authentication

### Evidence Authentication Method

For maximum legal credibility, digitally sign all evidence:

```bash
# 1. Generate private signing key (one-time)
openssl genrsa -out evidence-signing-key.pem 4096

# 2. Generate certificate (one-time)
openssl req -new -x509 -key evidence-signing-key.pem \
  -out evidence-signing-cert.pem -days 3650 \
  -subj "/C=US/ST=DC/L=Washington/O=FBI/CN=Cyber Division"

# 3. Sign evidence file
openssl dgst -sha256 -sign evidence-signing-key.pem \
  -out example-com-2026-05-31.sig \
  example-com-2026-05-31.png

# 4. Verify signature later
openssl dgst -sha256 -verify <(openssl x509 -in evidence-signing-cert.pem -pubkey -noout) \
  -signature example-com-2026-05-31.sig \
  example-com-2026-05-31.png
```

## FAQ

**Q: What if we forget to hash the evidence immediately?**  
A: Document the delay, note who discovered the gap, and hash immediately upon discovery. Explain the delay in your documentation. The chain of custody is not automatically "broken," but the gap weakens your evidence's reliability.

**Q: Can we modify evidence after collection?**  
A: No. Any modification must be documented with explanation, done by authorized personnel only, and justified by legal/investigative need. Better approach: work with copies, preserve original.

**Q: What's the minimum documentation needed?**  
A: Evidence ID, case reference, collector identity, timestamp, target URL, file hash, and signature. Better to document everything.

**Q: How long must we keep evidence?**  
A: Varies by jurisdiction and crime. Federal: typically 6+ years. State: varies. Consult with prosecutor. Document destruction authorization when applicable.

**Q: Can we use cloud storage for evidence?**  
A: Depends on jurisdiction and sensitivity. Better: Use local, controlled storage with encryption. If cloud necessary: Document security measures, encryption, access logs.
